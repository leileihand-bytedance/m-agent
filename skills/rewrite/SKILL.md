# 材料润色 Skill

## 核心定位

本 skill 只处理“用户直接贴出来的一段文字”的润色修改。

它不是：

- 直报写作
- 简报写作
- 审核纠错
- 联网补资料

如果用户要写直报、写简报、基于链接起草材料，应该交给对应 skill，不要混用。

## 输入范围

第一版只接受：

- 用户直接粘贴的原文
- 用户的润色要求，例如“更正式一点”“压缩一些”“顺一下逻辑”
- 本 skill 上一版输出的继续修改要求

用户直接粘贴原文时，当前支持两种常见说法：

- 要求在前，原文在后，例如“帮我润色这段：……”
- 原文在前，要求在后，例如“……原文全文\n\n帮我整体润色一下”

在独立材料润色 Bot 中，如果用户这一条只粘贴原文、没有说明修改方向，入口先保存原文并做一次简短追问，不直接调用模型。用户可补充具体要求，也可回复“按默认方式润色”；同一条消息已包含原文和要求、或是在继续修改本 Bot 上一版结果时，不重复追问。该入口状态只服务 `rewrite`，不能拿来组装直报、简报或综合调研任务。

第一版不读取：

- 网页链接
- Word/PDF
- 外部知识库和联网搜索
- 搜索结果

## 微众银行语料辅助

当用户原文明显与微众银行相关时，本 skill 可以通过底座受限工具读取少量高相关的本地微众银行信息库材料，用于核对：

- “微众银行”等机构名称
- “微业贷”“微粒贷”“微贸贷”等产品名称
- 信息库已有且依据唯一的标准表述

进入模型上下文的语料必须标记为 `material_role=verification_reference`，明确表示“复核参考”，不能作为普通写作素材。

语料只作为口径核对参考，不能扩大原文的事实边界：

1. 不把复核语料中原文没有的任何内容写入正文；复核通过不等于允许补充，尤其不能补入数据、日期、荣誉、案例或新观点。
2. 只有语料提供唯一、直接依据时，才可规范明显写错的专有名称。
3. 用户原文与语料、不同语料之间存在实质冲突时，不静默改数或自行选择口径，应请用户确认。
4. 普通材料不调用微众银行信息库，不注入微众银行表达。
5. 信息库不可用或没有高相关结果时，仍按普通润色规则处理，不声称已经核实。
6. 当前授权范围只覆盖已确认公开的现有语料；以后新导入的非公开或未分类材料不能自动沿用本次授权。

## 企业微信入口

独立材料润色 Bot 从注册表层只加载本 skill，不接直报、简报、综合调研或审核。该入口只接收直接粘贴文字，文件和网页链接会提示用户改为粘贴原文；它使用独立任务、待确认原文和会话目录，不继承原写作 Bot 的活跃稿件。待确认原文默认保留 30 分钟，Bot 重启后仍可接续，成功处理或用户取消后清除。

## 默认规则

如果用户只说“帮我润色一下”，默认遵循以下原则：

1. 只优化表达、语序、逻辑顺滑度和措辞正式度。
2. 不新增事实，不补新观点，不扩写没有依据的内容。
3. 如果原文明显没写完，例如句子断开、用了“XX”“某某”等明显占位，可做最小必要补全。
4. 如果无法确定该怎么补，设置 `needs_clarification=true`，请用户补充，不要硬编。

## 对话式修改

如果用户是在本 skill 的上一版基础上继续说：

- 再正式一点
- 第二段太硬
- 再短一点
- 保留原意

就按上一版继续改，不把它当成重新起草。

续改必须使用底座 `RevisionEngine` 的统一结构化范围。用户点名某段时，未点名段落由代码恢复；目标篇幅、数字保持和指定删除项在生成后检查，未通过时最多进行一次带反馈修正。本 skill 只保留润色的业务规则，不维护独立改稿解析器。

## 输出要求

结构化结果包含：

- `body`：润色后的正文
- `revision_note`：1-2 句修改说明
- `revision_plan`：底座内部追踪字段，不在正常用户回复中展示
- `needs_clarification`
- `message`

第一版 `title` 固定为空字符串，`sources` 固定为空列表。

## 执行要求

1. 保留用户原意，不擅自改变结论。
2. 不编造人名、机构、时间、数据、政策或事实。
3. 微众银行语料只能用于口径核对，不能作为扩写材料。
4. 修改说明只概括“改了什么”，不要重复整段正文。
5. 如果用户明确要求“保留原文风格”“不要改动太大”，优先做轻改。
6. 如果用户明确要求“更正式”“更规范”，可以提升书面化程度，但不能变成直报、简报体例。
