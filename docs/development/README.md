# M-Agent 后续开发总指南

本文档给 Codex、Claude Code 和后续开发者使用。目标是让后续开发可以通过自然语言持续推进，同时不破坏现有功能。

## 一句话结论

M-Agent 后续采用：

```text
M-Agent 自建底座
  + Pydantic AI 运行层
  + 文件化 skills
  + 受限工具层
  + 企业微信统一入口
```

它不是 Dify 这样的可视化平台，也不是 Hermes/OpenClaw 那种个人助手系统。它是一个代码型、可测试、适合 Codex/Claude Code 持续开发的业务 agent 底座。

## 当前重点

已经跑通的最小链路：

```text
本地 CLI / 本地 demo 输入自然语言
  -> 路由到 direct_report
  -> 创建独立任务目录
  -> 校验用户是否有 skill 权限
  -> 读取用户 URL
  -> Pydantic AI Agent 生成结构化结果
  -> 写入任务结果
  -> 输出标题、正文、来源
```

底座当前还具备：

- 直报 Bot 真实企业微信文本入口适配。
- 企业微信文本消息核心适配。
- 写作 Bot 短任务组装 v1：支持用户先发材料再说明用途，或先说明用途再连续发送多个材料，最后回复“开始写”；结构化任务需要澄清时保留原文件和会话，用户回答后自动续跑。
- 写作 Bot 文件消息接收 v1：可把企业微信 Word/PDF/PPTX 持久化暂存为结构化请求材料，Bot 重启后可恢复有效会话；后续仍需把组装接口下沉为底座公共能力。
- 网页读取，以及 DOCX/PDF/PPTX 统一文档读取工具；完整解析结果保存在任务 `work/`，扫描 PDF 可按需 OCR，PDF/PPTX 可显式逐页渲染。
- 用户/skill 权限策略。
- 共享用户名称映射表，写作 Bot 和审核 Bot 都能把企业微信 `userid` 对应到可读用户名。
- 每次请求独立 job 目录。
- 平台级会话状态和稿件版本链 v1。
- 开发期写作对话日志，可用于前期问题排查，稳定后可关闭。
- CLI 配置检查。
- 本机项目控制台：保留 Skill 开关、用户权限和最近任务，通过可缩放关系图和状态清单自动展示五层整体架构、功能关系、建设状态、六个板块进展、开放 TODO、最近 Git 更新、知识库/任务数量和 Bot 心跳。

## 当前进度快照

更新时间：2026-07-16

### 本轮 P1 底座交付的实际含义

2026-07-15 完成的是一套供所有 Skill 复用的生产底盘，不是新增某一个写作功能：

- **收材料**：写作和审核共用安全、可恢复的任务暂存内核，并统一 `wait`、`submit`、`cancel`、`bypass` 协议。
- **跑任务**：持久化任务执行器内核已具备消息幂等、排队、并发限制、取消、租约和进程重启恢复能力。
- **读文件**：扫描 PDF 可按需 OCR，PDF/PPTX 可在有需要时逐页渲染，并设置页数、时间、像素和容量上限。
- **发结果**：写作和审核共用附件交付能力，支持动态超时、完整重试、运维告警和“处理编号”兜底，不静默压缩图片。

必须区分当前生效范围：公共暂存、文档读取和附件交付已经接入现有入口；持久化执行器已经完成并通过自动化测试。当前真实 handler 包括纯文字、单个通用 Word、内参、半月报、公文格式、单个静态 HTML 和单份 PPTX 低级错误审核七类单项审核，以及直报、`writer1`、`writer2`。`research_synthesis` 和多文件联合审核尚未切换，因此仍不能表述为“所有 Bot 任务都能自动续跑”。下一步先完成这些已接入任务的真实企业微信故障验收。

当前整体状态：

- 审核 Bot 继续独立运行，已支持内参周报、半月报、通用 Word/文字审核、单个静态 HTML 文字审核、显式公文格式审核、单份 PPTX 低级错误审核和 `.docx` 多文件联合审核 v1，不急于迁移到统一入口。网页 PPT 的 HTML 审核意见按 `slide` 容器顺序显示页码，普通 HTML 显示段落位置；不计算浏览器打印分页。
- 七类单项审核已接入审核专用 SQLite 持久队列：文件到达时即时确认，单文件入队后不再重复发送队列或类型提示；HTML 和 PPTX 分别直接进入独立任务，不参与 Word 归集、格式审核或联合审核。HTML 只提取静态可见文字并复用通用审核，网页 PPT 结果保留 `slide` 页码；PPT 包不依赖其他业务审核引擎，只审可编辑文字、表格和可读图表内容，结果仅含页码、原文和代码生成的事实问题，不提供修改建议，不返回附件。`“帮我查一下格式”`等短口语要求会在文件到达前写入持久 intake；“也做一下文字审核”等文件后操作指令不会另建纯文字任务。同一消息幂等，处理完成后不重复调用模型，队列结果只发送一次，发送状态不确定时停止自动重发并通知用户和运维，worker 异常会告警并自动重启。后台任务编号用于恢复和排障，正常受理时不向用户展示；只有异常需要人工介入时才显示“处理编号”。Word 单文件仍可在 30 分钟内追加独立格式审核；多文件联合审核暂未切流。
- 直报、`writer1`、`writer2` 已接入独立写作 SQLite 持久队列：入队前创建正式 job 并冻结 URL/文字/文件快照，后台按处理、会话收尾、主动发送三个阶段记录检查点；同一消息幂等，已完成 job 不重复调用模型或增加稿件版本，发送状态不确定时停止自动重发并通知用户和运维。
- 写作入口已增加在途任务保护和消息去重：同一用户初稿未完成时暂不接收下一批材料或改稿；重复文件不会把 `writer1` 错切为 `writer2`。待澄清材料会恢复，用户确认继续后可忽略失败链接并使用成功读取的素材写作。
- 通用审核真实质量评测工具已落地：历史文件按文件内容和正文近重复双重去重，首批 5 份覆盖问卷、附件引用、超长文、表格和常规材料；真实结果留在 `M-Agent-Files/evaluations/`，仓库只保留工具、测试和脱敏说明。
- 首轮人工评分确认 46 条候选的自动标注位置全部准确；英文 `People’s`、`China’s`、`people‘s` 等弯撇号被 `quote-pair` 误当中文单引号的问题已修复，真正缺失的中文引号仍会保留。
- 首轮及后续实测误报已采用脱敏定向回放修复：错别字提示词纠正为“原文错误片段应为正确写法”，证据矛盾候选进入定向复核；模型生成的错别字候选现在无论篇幅都结合完整句子和前后文做两次严格复核，只有一致确认才保留，避免把相邻但分属不同语法成分的字拼成错词。分块不再拆开“数字序号 + 短标签 + 后续说明”，从根因消除 `Fortune` 块尾误报，而非泛化过滤短英文。原始 Word 中真实存在的 `、 ` 不删除意见，而是精确标记并说明为顿号后多余空格。
- 格式审核支持两种消息顺序：先说格式审核再发文件，或先发文件后在 30 分钟内补说格式审核；待处理状态持久化，Bot 重启后可恢复。
- 用户直接发 1 份 `.docx` 时自动走原单文件审核，连续发 2 至 5 份时无需“联合审核”或“开始审核”指令，自动归为联合审核；文件到达即确认，后台默认使用 8 秒静默窗口分流。
- 联合审核不按上传顺序默认正文；系统综合文件名和正文附件引用识别主文件，证据仍不唯一时才要求用户指定，再执行逐文件审核、附件确定性检查和双边原文证据的跨文件语义检查。
- 通用审核已具备代码规则、分段语义审核、`200` 至 `100000` 字通篇逻辑校对、长文双扫描候选复核和内容匹配优先的 Word 标注流程。

## 不要做的事

- 不要把所有能力写进一个大 prompt。
- 不要让模型自由决定调用任意工具。
- 不要在 skill 中直接读取 `.env`。
- 不要让外部用户访问本机任意文件。
- 不要一上来重写 `app/review/` 或 `app/writing/`。
- 不要为了跑 demo 把密钥写进代码。
