# 月度开发日志机制

M-Agent 的开发过程必须完整记录，但不再把所有历史堆入根目录一个大文件。

## 存储位置

```text
STATUS-REPORT.md                                    # 本机短索引，不进入 Git
M-Agent-Files/runtime/development-logs/YYYY-MM.md  # 每月完整开发日志，不进入 Git
```

根目录索引只说明日志位置和已有月份；开发记录正文全部写入对应月份文件。

日志目录默认跟随 `M_AGENT_DATA_DIR`；未配置时使用项目同级的 `M-Agent-Files/`。只有特殊部署才单独设置 `M_AGENT_STATUS_REPORT_DIR` 覆盖月度日志目录。

## 每条记录

每个完成并验证、且已经同步远端的逻辑开发节点记录一条：

1. 完成功能：完成了什么或解决了什么问题。
2. 能力变化：实际改变了什么流程、能力或用户体验。
3. 关键验证：自动化测试、真实链路、人工检查或故障测试结论。
4. 当前边界/下一步：仍未完成什么、有什么风险、接下来做什么。
5. 影响模块和 Git 哈希：只用于分类与技术追溯。

禁止把文件列表、文件数量、提交摘要和推送范围作为日志主体。

## 安全边界

月度日志不得包含：

- 用户材料、业务正文和真实文件名。
- 真实用户 ID、Bot ID、密钥和环境变量值。
- 错误堆栈、PID、本机任务路径和临时日志路径。
- 未脱敏的模型输入输出。

需要排障的技术细节进入受限运行日志；开发日志只写脱敏结论和处理效果。

## 自动生成

禁止直接 `git push`。受管推送命令在远端成功后才写当月日志：

```bash
uv run --locked python scripts/project_docs.py push \
  --summary "完成了什么功能" \
  --impact "实际改变了什么能力" \
  --verification "做了哪些关键验证" \
  --next-step "当前边界或下一步"
```

- post-commit 只提醒本地是否尚未推送，不写日志。
- pre-push 再次执行文档闸门并阻止绕过受管推送。
- 推送失败、远端分叉或验证未完成时，不能写成已完成。
- 同一远端提交使用唯一标记去重，重复执行不会生成两条记录。

## 历史迁移

旧版根目录 `STATUS-REPORT.md` 可以通过以下命令按标题日期拆分：

```bash
uv run --locked python scripts/project_docs.py migrate-status-report
```

迁移要求：

- 先按 `YYYY-MM` 生成月度文件，再校验标题记录总数一致。
- 原记录按原顺序和原文保留，不在迁移时改写历史结论。
- 无法识别月份的内容写入单独遗留文件，不得静默丢弃。
- 校验通过后才把根目录文件替换为索引。
- 命令可重复执行，不得重复写入同一记录。

## 事实优先级

1. 当前代码和自动化测试。
2. 当前架构、模块 README、能力和 Skill 文档。
3. 当前 TODO。
4. Git 历史。
5. 月度开发日志。
6. 历史设计和计划。

月度日志保证过程完整，但不能代替当前事实文档。
