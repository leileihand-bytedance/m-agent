# M-Agent 待办管理

本文档是 M-Agent 当前待办事项的统一入口。Codex、Claude Code 和后续 AI 开发工具进入项目后，必须在阅读架构文档后阅读本文档。

## 文档定位

```text
docs/development/TODO.md      # 当前还没完成、需要后续推进的事项
Git 提交和核心文档            # 已经完成的事实依据
STATUS-REPORT.md              # 本机自动开发日志，不进入 Git
docs/development/README.md    # 总指南和文档地图
```

项目运行环境已统一为 uv 管理的 Python 3.13.14、根目录 `.venv` 和 `uv.lock`；这是已完成的交付基线，不作为待办重复维护。所有后续待办的测试和脚本都使用 `uv run --locked ...`。

规则：

1. 新增待办时，优先写入本文档，不要只写在聊天记录、状态报告或零散文档里。
2. 完成待办后，更新本文档状态、对应核心文档和测试；Git hook 会自动写本机状态摘要。
3. 如果待办已经失效，标记为 `已取消` 或 `已暂缓`，并写明原因。
4. 不把已完成历史长期堆在本文档；完成细节写入对应模块文档和 Git 提交。
5. 涉及具体实现时，仍必须同步更新对应模块文档、测试和必要的 skill 文档。

## 状态规则

待办状态只使用以下几类：

```text
未开始      # 已确认有价值，但还没动手
进行中      # 正在开发或已经有部分实现
已暂缓      # 方向成立，但暂时不做
已完成      # 已实现、已验证、已同步核心文档
已取消      # 不再做
```

优先级只使用以下几类：

```text
P0  # 影响当前可用性或生产测试
P1  # 近期主线能力
P2  # 重要但不阻塞近期使用
P3  # 可观察、可优化、可后续讨论
```

### TODO-028：支持审核意图补充、任务追加与显式切换

状态：未开始

优先级：P2

归属：审核入口 / 底座任务执行 / 多轮交互

背景：当前用户直接上传单个文件、没有提前说明审核类型时，系统会按默认规则启动通用内容审核；内容审核启动后会保留最近原件 30 分钟。用户随后补充“再帮我查一下格式”等要求时，系统已经能够复用该文件，创建一项独立公文格式审核，原内容审核继续执行。这保证了默认场景可用，但也存在一个后续交互问题：用户可能只是先上传文件，再补充“帮我查一下格式”，实际只想做格式审核，此时却会先后收到通用审核和格式审核两份文档。当前尚未完成“文件后补充意图”与“追加第二项审核”的语义区分，也不支持用户明确说“不要审内容了，改成只审格式”后取消或替换已经入队、运行中的内容审核。该问题暂不影响当前默认通用审核，待专项设计和回归后再修改生产行为。

目标：

- 保留并验收现有追加能力：内容审核已经入队或运行后，用户说“再审一下格式”“刚才的文件也查一下格式”，复用最近原件创建独立格式审核，原内容审核不受影响，两项结果分别交付。
- 将“也做一下文字审核”“再做一下内容审核”等完整短句识别为对文件的操作指令，不得把指令文字本身建立为纯文字审核任务；真实正文和包含正常陈述内容的长句不能被误拦截。
- 增加文件后的短暂“意图补充窗口”：用户没有提前说明类型、先上传文件后立即补充“帮我查一下格式”等要求时，优先把它理解为对刚才文件审核类型的补充，避免过早启动默认通用审核。窗口结束且没有补充要求时，仍自动执行默认通用审核，不增加每次都要用户确认的负担。
- 明确三种意图：`“只查格式”“不用审文字”`表示仅格式或切换；`“也查一下格式”“再做一个格式审核”`表示追加；文件上传后在意图补充窗口内单独说`“查一下格式”`，优先按仅格式处理。超过窗口、原任务已经运行且表述不明确时，Bot 应先追问一次，不能擅自取消，也不能直接返回两份用户并不需要的文档。
- 明确切换意图：只有“不要审内容了”“改成只审格式”“取消刚才的内容审核”等显式表达，或意图补充窗口内确定的仅格式要求，才进入替换流程；含“也”“再”的追加表达不能取消原任务。
- 区分任务状态：原任务仍在 `queued` 时可以原子取消并创建格式任务；已经 `running` 时不得静默切换，先告知内容审核已经开始，并让用户确认是继续追加格式审核还是尝试取消原任务；已经完成时只能追加新任务。
- 复用后台任务编号做状态关联，但普通用户不需要输入或看到编号；系统按当前用户最近一项可切换审核任务自动定位，存在歧义时再用文件名和任务类型让用户选择。
- 保证竞态安全：用户确认、worker 领取任务和进程重启同时发生时，取消只能成功一次；已取消任务不得继续交付结果，已经确认交付的结果不得撤回或重复发送。
- 保留文件安全边界：追加或切换只能复用该用户最近审核任务冻结的输入副本，默认有效期 30 分钟，不能读取其他用户或本机任意文件。

验收标准：

- 真实企业微信中，文件进入内容审核后回复“再帮我查一下格式”，原内容审核和新增格式审核分别完成并返回结果。
- 真实企业微信中先上传单个文件、随后在意图补充窗口内回复“帮我查一下格式”，最终只执行并返回一项格式审核，不再额外返回默认通用审核文档。
- 直接上传单个文件且在意图补充窗口内没有提出其他要求时，窗口结束后仍自动执行一次默认通用审核，不漏任务、不要求用户再次确认。
- 文件后回复“也帮我查一下格式”时仍明确视为追加，内容审核和格式审核各返回一项结果；`“只查格式”`与`“也查格式”`不得产生相同任务组合。
- 原任务仍在排队时回复“不要审内容了，改成只审格式”，原任务进入 `cancelled` 且不调用审核引擎、不发送结果，只生成一项格式审核。
- 原任务已经运行时，Bot 不擅自取消或重复建任务，而是返回一次状态说明和明确选择；用户确认后只执行所选动作。

- Bot 在取消或追加过程中重启后，任务状态、文件关联和用户选择可以恢复，不会永久停留在处理中。
- 自动化测试覆盖意图补充窗口内仅格式、无补充时默认通用审核、明确追加、排队中切换、运行中确认、完成后追加、重复指令、过期文件、进程重启和多用户隔离。
