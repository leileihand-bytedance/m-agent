## 当前待办

### TODO-033：补齐企业微信结果交付确认与安全恢复

状态：进行中

优先级：P0

归属：底座 / 企业微信入口 / 测试和交付

背景：当前持久任务会在调用企业微信主动发送前把交付项标为 `sending`。如果 SDK 调用超时或抛出异常，系统只能知道“已经发起发送尝试”，无法区分请求尚未到达企业微信、企业微信已经接收但本机未收到确认两种情况。现有保护机制会停止自动重发，避免重复发送，但已生成的正文或附件可能停在“送达未知”，用户既收不到结果，也未必能收到同一通道发送的失败提示。

当前进展：已核实项目使用的 `wecom-aibot-sdk 1.0.7`。主动发送会等待与本次请求关联的 SDK 回执：`errcode=0` 可确认送达，非零回执可确认未送达，等待超时或无法解释的发送异常只能标记为送达未知；该版本没有可用的发送状态查询接口，因此不能在超时后自动向企业微信反查。底座已统一文本和附件的三态检查点，保存本地交付尝试编号、判断依据、时间和脱敏回执关联标识；明确未送达可按原检查点重发，送达未知必须由本机管理员确认后才能重发、确认已送达或关闭，均不重新调用模型。自动化回归已覆盖主要分支，剩余工作是使用专用测试 Bot 完成真实超时、拒绝、重启和人工恢复验收。

目标：

- 调研并接入企业微信 SDK 可提供的发送回执、请求标识或状态查询能力；若平台不支持查询，明确记录可追溯的“送达未知”状态和人工恢复流程，不能把它伪装成已失败或已送达。
- 将交付状态至少区分为“明确未送达”“已确认送达”“送达未知”，并保存判断依据、时间和必要的脱敏关联标识。
- 对可确认未送达的任务，只重发原任务检查点中的正文或附件，不重新调用模型、不重新解析材料；对已确认送达的任务禁止重发。
- 为送达未知任务提供受权限控制的恢复操作：先查询或人工确认，再以原任务结果完成交付或关闭任务，全部过程进入脱敏运维事件。
- 文本和附件共用同一套状态语义；故障提示的发送失败不能覆盖或误判原结果的交付状态。

验收标准：

- 模拟本地校验失败、明确的企业微信拒绝、发送超时和“远端已接收但本机确认丢失”四类情况时，系统能进入正确状态，不把它们全部归为同一个安全错误码。
- 已生成但明确未送达的简报、直报、审核结果或附件，可在不重新调用模型的前提下按原检查点恢复交付一次。
- 已确认送达的结果在 Bot 重启、重复消息和人工恢复后均不会重复发送。
- 送达未知任务不会静默结束；运维侧能看到处理依据和恢复入口，用户侧在通道可用时能收到简洁、可理解的状态提示。
- 自动化测试覆盖文本、附件、任务重启、重复消息、超时、SDK 异常、SDK 无状态查询和人工恢复分支；再使用脱敏测试 Bot 完成至少一轮真实企业微信故障验收。

### TODO-032：研究并重构下一代内容生产模式

状态：进行中

优先级：P3

归属：内容生产方法论 / 底座 / 内容生产 Skills

背景：当前 M-Agent 的主要建设方式，是把已有的人工流程拆成“收集材料、搜索、写初稿、审核、改稿和交付”，再由 AI 和代码提高速度、降低重复劳动。这一阶段有明确价值，但本质上仍是在自动化原来的工作方式。等这些基础流程稳定后，如果继续只优化提示词、模型或单个 Skill，成稿效果和工作价值可能逐渐遇到上限。项目需要长期研究：AI 参与后，内容为什么生产、生产什么、怎样形成判断、怎样与用户协作，以及成果如何反哺下一次任务，是否可以采用不同于原有流程的新模式。

研究目标：

- 从“替代人工步骤”转向“人机共同完成研究、判断、策划、写作和复盘”，明确哪些环节适合自动化，哪些判断必须由用户或编辑确认。
- 重新定义内容生产的起点：不只接收一批材料和一个文种，还能理解写作目的、受众、使用场景、核心问题、希望促成的行动和不可突破的口径。
- 研究内容生产的中间层：先形成事实台账、证据关系、关键判断、叙事策略和结构方案，再生成正文，避免所有复杂工作都压在一次模型写作中。
- 建立可持续的反馈闭环：把用户改稿、最终采用版本、未采用原因、常见口径和高质量表达沉淀为可复用资产，而不是每次从头生成。
- 研究从“被动接单”扩展到“主动辅助”的合理边界，例如选题发现、政策和行业变化提醒、采访问题生成、同一事实面向不同受众的版本规划，以及已完成材料的知识再利用。
- 建立结果导向的质量衡量方式，不只统计完成数量和处理时间，还观察事实可靠性、观点和洞察、受众适配度、用户修改幅度、最终采用情况和内容资产复用价值。
- 在全过程保持权限、保密、来源追溯和人工确认边界，不能为了提高自动化程度而扩大材料暴露范围或让模型替代责任判断。

研究方式：

- 第一阶段先观察现有直报、简报、综合调研、深银协动态和材料润色的真实使用过程，记录最耗时间、最依赖人工判断、最容易反复改稿和最影响最终采用的环节。
- 第二阶段选择一至两个高频场景做小规模对照试验，对比“直接生成正文”和“目标澄清—事实台账—内容策划—生成—反馈沉淀”等不同模式的质量、耗时和用户修改量。
- 第三阶段只把已经证明有效的环节沉淀为底座公共能力，例如内容任务简报、证据与主张结构、编辑决策记录、用户反馈资产化和效果评估；业务口径继续留在各 Skill。
- 定期复盘研究结论，并允许得出“不需要整体重构、只需调整部分环节”的结论；不预设必须引入 LangGraph、多 Agent、某个新框架或新的模型供应商。

阶段产出：

- 一份基于真实任务的现状流程图和主要瓶颈清单，区分工具问题、模型问题、材料问题、流程问题和业务判断问题。
- 一套下一代内容生产模式的设计原则、目标流程和人机职责边界。
- 一至两个可验证的小规模试验方案，以及与现有流程可比较的质量和效率指标。
- 一套把用户改稿和最终采用结果转化为质量回归样本、知识资产和后续优化依据的方法。
- 对底座、知识库、Skill 和管理后台需要新增或调整的能力形成迁移建议，明确哪些可以渐进演进，哪些不值得建设。

进入实施阶段的条件：

- 当前持久任务、文件读取、交付、告警等生产底座已稳定，不再被基础可靠性问题持续干扰。
- 已积累一批可追溯的真实任务、用户改稿和最终版本，能够用事实识别瓶颈，而不是只凭主观设想设计新流程。
- 至少一个小规模试验能证明新模式在成稿质量、用户修改量、完成时间或内容复用价值上明显优于现有方式。
- 新模式能在不破坏现有 Skill、入口和安全边界的前提下渐进接入，并有明确的回退方案。

### TODO-031：统筹审核规则，建设共享审核核心和可组合规则包

状态：进行中

优先级：P1

归属：审核 / 测试和交付

背景：当前纯文字、通用 Word 和 HTML 已复用通用审核引擎，内参、半月报、PPT、公文格式和多文件审核分别维护专属规则；各链路又不同程度重复实现问题结构、模型调用、原文证据校验、候选复核、去重和定位。后续继续增加通用规则时，如果在每个审核器复制一份，会造成规则口径和回归测试分叉；但如果让所有材料直接调用现有 `general_reviewer.py`，又会把 Word 文章假设带入标题、表格、HTML、PPT 和专属材料，增加误报与耦合。

目标：

- 先建立完整规则矩阵，把现有规则标为“所有材料通用”“部分材料条件通用”或“类型专属”，同时记录适用对象、原文证据要求、定位方式、返回方式和当前测试样本。
- 抽取格式无关的共享审核核心：统一问题结构、模型调用与重试、逐字原文证据校验、上下文复核、候选去重、耗时与调用指标；共享核心不包含内参、半月报、PPT或公文格式业务规则。
- 建立单一规则来源和可组合规则包。通用规则的定义、提示要求和基础测试只维护一份，各审核类型通过静态审核配置选择启用规则和适用条件，专属规则继续留在各自模块。
- 一次审核按当前材料组装“通用规则 + 专属规则”后执行，不把每条规则拆成独立模型调用，不引入多 Agent、LangGraph 或动态插件编排。
- 第一批只统筹当前已经复用同一引擎的纯文字、通用 Word 和 HTML；稳定后再迁移内参和半月报。PPT 只复用问题结构、证据和运行核心，不直接复用 Word 提示词；公文格式和多文件跨文档规则保持专属。
- `正文不完整`、重复、编号、名称不一致和跨段逻辑等结构敏感规则必须保留材料适配条件，不能因为名称相同就全局启用。

验收标准：

- 每条通用规则只有一个权威定义和一组共享测试；审核类型只维护启用范围、阈值或确有必要的专属补充，不复制整条规则和提示词。
- 迁移前后规则 ID、用户可见问题、原文证据、标注位置和返回形式保持兼容；未迁移审核器继续按原流程工作。
- 同一份材料的模型调用次数不得因规则拆分而增加；提示词只装入当前材料启用的规则，不能形成包含所有业务规则的大提示词。

### TODO-028：支持审核意图补充、任务追加与显式切换

状态：未开始

优先级：P2

归属：审核入口 / 底座任务执行 / 多轮交互

背景：当前用户直接上传单个文件、没有提前说明审核类型时，系统会按默认规则启动通用内容审核；内容审核启动后会保留最近原件 30 分钟。用户随后补充“再帮我查一下格式”等要求时，系统已经能够复用该文件，创建一项独立公文格式审核，原内容审核继续执行。这保证了默认场景可用，但也存在一个后续交互问题：用户可能只是先上传文件，再补充“帮我查一下格式”，实际只想做格式审核，此时却会先后收到通用审核和格式审核两份文档。当前尚未完成“文件后补充意图”与“追加第二项审核”的语义区分，也不支持用户明确说“不要审内容了，改成只审格式”后取消或替换已经入队、运行中的内容审核。该问题暂不影响当前默认通用审核，待专项设计和回归后再修改生产行为。

目标：

- 保留并验收现有追加能力：内容审核已经入队或运行后，用户说“再审一下格式”“刚才的文件也查一下格式”，复用最近原件创建独立格式审核，原内容审核不受影响，两项结果分别交付。
- 将“也做一下文字审核”“再做一下内容审核”等完整短句识别为对文件的操作指令，不得把指令文字本身建立为纯文字审核任务；真实正文和包含正常陈述内容的长句不能被误拦截。
- 增加文件后的短暂“意图补充窗口”：用户没有提前说明类型、先上传文件后立即补充“帮我查一下格式”等要求时，优先把它理解为对刚才文件审核类型的补充，避免过早启动默认通用审核。窗口结束且没有补充要求时，仍自动执行默认通用审核，不增加每次都要用户确认的负担。
- 明确三种意图：`“只查格式”“不用审文字”`表示仅格式或切换；`“也查一下格式”“再做一个格式审核”`表示追加；文件上传后在意图补充窗口内单独说`“查一下格式”`，优先按仅格式处理。超过窗口、原任务已经运行且表述不明确时，Bot 应先追问一次，不能擅自取消，也不能直接返回两份用户并不需要的文档。
- 明确切换意图：只有“不要审内容了”“改成只审格式”“取消刚才的内容审核”等显式表达，或意图补充窗口内确定的仅格式要求，才进入替换流程；含“也”“再”的追加表达不能取消原任务。
- 区分任务状态：原任务仍在 `queued` 时可以原子取消并创建格式任务；已经 `running` 时不得静默切换，先告知内容审核已经开始，并让用户确认是继续追加格式审核还是尝试取消原任务；已经完成时只能追加新任务。
- 复用后台任务编号做状态关联，但普通用户不需要输入或看到编号；系统按当前用户最近一项可切换审核任务自动定位，存在歧义时再用文件名和任务类型让用户选择。
- 保证竞态安全：用户确认、worker 领取任务和进程重启同时发生时，取消只能成功一次；已取消任务不得继续交付结果，已经确认交付的结果不得撤回或重复发送。
- 保留文件安全边界：追加或切换只能复用该用户最近审核任务冻结的输入副本，默认有效期 30 分钟，不能读取其他用户或本机任意文件。

验收标准：

- 真实企业微信中，文件进入内容审核后回复“再帮我查一下格式”，原内容审核和新增格式审核分别完成并返回结果。
- 真实企业微信中先上传单个文件、随后在意图补充窗口内回复“帮我查一下格式”，最终只执行并返回一项格式审核，不再额外返回默认通用审核文档。
- 直接上传单个文件且在意图补充窗口内没有提出其他要求时，窗口结束后仍自动执行一次默认通用审核，不漏任务、不要求用户再次确认。
- 文件后回复“也帮我查一下格式”时仍明确视为追加，内容审核和格式审核各返回一项结果；`“只查格式”`与`“也查格式”`不得产生相同任务组合。
- 原任务仍在排队时回复“不要审内容了，改成只审格式”，原任务进入 `cancelled` 且不调用审核引擎、不发送结果，只生成一项格式审核。
- 原任务已经运行时，Bot 不擅自取消或重复建任务，而是返回一次状态说明和明确选择；用户确认后只执行所选动作。
- 同一切换指令重复投递不会重复取消、重复创建格式任务或重复发送结果。
- Bot 在取消或追加过程中重启后，任务状态、文件关联和用户选择可以恢复，不会永久停留在处理中。
- 自动化测试覆盖意图补充窗口内仅格式、无补充时默认通用审核、明确追加、排队中切换、运行中确认、完成后追加、重复指令、过期文件、进程重启和多用户隔离。

### TODO-018：改稿精准执行和版本链

状态：进行中

优先级：P2

归属：底座 / 直报 / 简报

背景：平台已有上一稿识别和版本链，但具体修改要求仍主要交给模型整体重写。对“只改标题”“删掉某句”“压短某段”等口语化局部要求，执行结果还不够精准，需要补充结构化修改动作和执行自检。

目标：

- 增加结构化“改稿操作解析”，先把用户要求拆成可执行动作。
- 支持常见改稿动作：压缩篇幅、扩写背景、调整标题、强化/弱化政策背景、删除指定内容、调整语气、保留事实但改变结构、突出某一主题。
- 改稿后做轻量自检，判断是否真的执行了用户要求，未执行时自动再修一次或提示用户需要更明确指令。

当前基线：平台已经保存版本链，并支持“回到上一版”“按第一版再改”“参考第 2 版”等版本选择。

验收标准：

- 用户要求“只改标题”时，正文原则上不大改。
- 用户要求“删掉某句/某段”时，输出中不再出现对应内容。
- 用户要求“压缩到约700字”时，正文长度应明显收敛到目标区间。
- 用户要求“强化政策背景”时，政策背景应出现在开头或主体前部，而不是只在结尾泛泛带过。
- 用户要求“不要改变事实和数据”时，原有关键事实、数字和来源不得被随意改写。
- 任务记录中能追溯每个版本对应的修改要求和父版本。
- 自动化测试覆盖标题-only 修改、删句、压缩篇幅、强化政策背景、版本链记录。
