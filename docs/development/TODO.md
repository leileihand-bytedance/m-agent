# M-Agent 待办管理

本文档是 M-Agent 当前待办事项的统一入口。Codex、Claude Code 和后续 AI 开发工具进入项目后，必须在阅读架构文档后阅读本文档。

## 文档定位

```text
docs/development/TODO.md      # 当前还没完成、需要后续推进的事项
Git 提交和核心文档            # 已经完成的事实依据
STATUS-REPORT.md              # 本机自动开发日志，不进入 Git
docs/development/README.md    # 总指南和文档地图
```

规则：

1. 新增待办时，优先写入本文档，不要只写在聊天记录、状态报告或零散文档里。
2. 完成待办后，更新本文档状态、对应核心文档和测试；Git hook 会自动写本机状态摘要。
3. 如果待办已经失效，标记为 `已取消` 或 `已暂缓`，并写明原因。
4. 不把已完成历史长期堆在本文档；完成细节写入对应模块文档和 Git 提交。
5. 涉及具体实现时，仍必须同步更新对应模块文档、测试和必要的 skill 文档。

## 状态规则

待办状态只使用以下几类：

```text
未开始      # 已确认有价值，但还没动手
进行中      # 正在开发或已经有部分实现
已暂缓      # 方向成立，但暂时不做
已完成      # 已实现、已验证、已同步核心文档
已取消      # 不再做
```

优先级只使用以下几类：

```text
P0  # 影响当前可用性或生产测试
P1  # 近期主线能力
P2  # 重要但不阻塞近期使用
P3  # 可观察、可优化、可后续讨论
```

## 统筹视图（2026-07-13）

当前下一阶段主线调整为“审核质量产品化”。现有审核 Bot 已能稳定运行，内参、半月报和通用审核也都有专项规则；独立公文格式审核已提前完成，当前重点仍是用真实文件建立可重复验证的内容审核质量基线，再按依赖顺序补齐多文件和 PPT。

推荐推进顺序：

| 阶段 | 待办 | 本阶段目标 | 进入下一阶段的条件 |
|---|---|---|---|
| 立即推进 | `TODO-023` | 建立真实文件回归集，稳定通用审核的误报、漏报、长文逻辑和标注位置 | 已知问题都有脱敏回归样本；连续真实测试不再反复出现同类回归 |

| 第三阶段 | `TODO-024` -> `TODO-021` | 在统一文档服务之上做正文与附件联合审核 | 多个文件不会被拆成多个任务；主附件关系和跨文件问题可验证 |

- 直报和简报当前保持可用，`TODO-001`、`TODO-002` 改为真实样本驱动的持续优化，不再和审核主线同时做大规模重构。

### TODO-024：建设底座统一文档服务

状态：未开始

优先级：P1

归属：底座 / 企业微信入口

背景：当前 Word/PDF 读取以面向单个写作 skill 的轻量文本提取为主，PDF 只返回截断文本，尚未统一保留页码、内容块、样式、解析告警和扫描件识别结果；PPT 也还没有公共解析基础。如果继续分别增加 `pdf_reader`、`ppt_reader`，写作、审核和知识库会重复处理文件安全、长文分块、位置定位和中间产物管理。

目标：

- 在 `app/platform/` 建立统一文档服务，统一管理格式识别、解析器注册、标准文档结构、长文分块、渲染和可选 OCR 接口。
- 完整解析结果保存在任务 `work/` 目录，skill 按需获取材料片段；不得再通过统一截取前若干字符代替长文处理。
- PDF 第一阶段按页保留文字、页码、解析告警和扫描页判断；OCR 作为按需兜底，不默认处理所有页面。
- 为后续 `.pptx` 保留幻灯片号、对象位置、样式、图片和渲染结果接口，具体 PPT 审核规则仍由 `TODO-020` 承接。
- HTML 文件支持已明确暂缓，不进入本阶段格式白名单、近期实现范围和验收标准；现有网页链接读取能力保持不变。
- 文件只允许在本次任务目录内解析，拒绝宏格式、伪造类型、异常压缩包和任务目录外路径。
- 用户上传文件、待组装文件、解析 JSON、渲染图片、OCR 结果和系统生成文件全部保存在桌面 `M-Agent-Files/`，不得写回 Git 仓库。

存储约定：

```text
M-Agent-Files/runtime/intake/                         # 尚未明确用途或尚未开始处理的短期暂存文件
M-Agent-Files/tasks/<scope>/YYYY/MM/<job_id>/input/  # 用户原始文件
M-Agent-Files/tasks/<scope>/YYYY/MM/<job_id>/work/   # 解析、渲染、OCR 等中间产物
M-Agent-Files/tasks/<scope>/YYYY/MM/<job_id>/output/ # 返回给用户的报告、标注或生成文件
```

验收标准：

- Word、原生文字 PDF、扫描 PDF 和后续 PPT 使用同一套文件引用、任务隔离和标准文档接口。
- 长 PDF 可以按页和内容块供 skill 使用，完整解析结果不因模型上下文限制而在存储阶段丢失。
- 解析结果能定位到页码或幻灯片号，并明确记录未读取内容、OCR 使用情况和失败原因。
- Bot 重启后，尚在有效期内的待组装文件仍可恢复；过期文件按保留策略清理。
- 所有真实文件和中间产物都位于 `M-Agent-Files/`，仓库只保留脱敏自动化测试样本。
- 自动化测试覆盖格式伪造、路径越界、超限文件、异常压缩包、长文分块、扫描页判断和任务隔离。

### TODO-023：建立通用审核真实样本回归和质量基线

状态：进行中

优先级：P0

归属：审核 / 测试和交付

背景：通用审核已经具备分段文字审核、通篇逻辑校对、长文候选复核和“内容匹配优先、段号保底”的标注流程。2026-07-13 审核专项自动化测试共 `200` 项通过，但现有测试以合成样本和固定规则为主；真实长文测试仍出现过误报、漏报和标注位置不准确，说明还缺少可持续复测的真实质量基线。

目标：

- 从经授权的真实文件中建立短文、常规长文、超长文、表格密集文档和附件引用文档样本集。
- 每个样本记录人工确认的“应报问题、不应报内容、错误原文、正确位置和修改建议”，同时记录模型和代码实际结果。
- 建立误报率、漏报情况、自动定位正确率、人工可搜索定位率、耗时和模型调用次数等指标。
- 用户发现新的误报、漏报或标错位置后，先增加脱敏回归样本和自动化测试，再调整规则、候选复核或定位逻辑。
- 真实用户原件、用户 ID 和敏感内容不进入 Git；仓库只保留脱敏片段、合成等价样本或不含敏感信息的授权样本。

验收标准：

- 至少覆盖普通文章、长文、表格、问卷、附件引用和日期/数字逻辑六类真实场景。
- 当前已知的标题误判、短重复误报、数字口径误报、附件编号名称错配漏报、段号漂移和短词定位错误均有回归测试。
- 有明确的基线报告，能比较每次修改前后的有效问题数、误报数、漏报数、定位结果和耗时，而不是只凭一次使用感受判断。

- 审核专项离线自动化测试持续通过，真实模型端到端失败时能区分模型/网络失败与规则回归。
