# M-Agent 本机管理后台

本文档说明当前管理后台的定位、能力边界和后续开发规则。

## 定位

管理后台是 M-Agent 的本机控制台，不是对外服务平台。

当前用途：

- 通过交互式关系图和状态清单查看项目总览，包括用户入口、通用底座、业务功能、工具知识库、运维数据五层架构、能力关系和建设状态。
- 查看底座、写作、审核、知识库、入口运维、管理后台六个板块的当前情况、最近更新和首要待办。
- 查看写作 Bot、审核 Bot、材料润色 Bot、运维 Bot 的最近心跳，识别正常、超时或未运行。
- 直接读取 `docs/development/TODO.md`，按优先级展示当前开放待办。
- 查看已安装的 `skills/*`。
- 开启或关闭某个 skill。
- 按需配置企业微信用户可以使用哪些 skill；默认页面不加载用户权限。
- 按需查看最近任务记录摘要；默认页面不加载任务记录。
- 后续管理政策知识库底层数据，包括查看、筛选、勾选删除和备份；该能力尚未完成。

默认只监听：

```bash
127.0.0.1
```

不要把它直接暴露到公网或公司内网。

## 启动

```bash
uv run --locked python -m app.admin.server --port 8787
```

打开：

```text
http://127.0.0.1:8787
```

## 当前能力

### 项目总览

控制台首页直接以五层架构关系图作为“项目总览”，不再显示“已启用能力、开放待办、累计任务、Git 状态”四个独立指标卡片：

```text
用户入口
  -> 通用底座
  -> 业务功能
  -> 工具与知识库
  -> 运维与数据
```

页面提供两个视图：

- `关系图`：展示能力节点之间经过代码维护的调用、材料提供、状态上报和结果回传关系；支持缩放、拖动、适应画布和点击节点查看详情。
- `状态清单`：按五层分组完整展示节点，作为详细扫描和关系图组件异常时的备用视图。

两个视图共用同一份服务端数据和状态筛选，不另建手工进度表。每个节点展示能力名称、当前建设状态、简要职责、事实依据和关联待办的下一动作。页面可以按以下状态筛选：

- `稳定运行`：能力已交付并作为当前稳定基础使用，或关联建设待办已经完成。
- `已上线·优化中`：主体能力已经可用，但仍有质量、口径或体验类开放待办。
- `建设中`：关联待办正在进行；对于明确配置的过渡能力，代码已经存在但待办仍未完成时也显示建设中。
- `待建设`：待办尚未开始，且没有对应已开始实施的证据。
- `已暂缓`：关联待办明确标记为已暂缓。
- `已关闭`：对应 Skill 已安装，但关联 Skill 当前全部关闭。

架构节点、明确关系、节点职责和允许作为事实依据的代码路径维护在 `app/admin/services.py`。新增、删除或改名能力节点时，必须同步更新关系表并通过“所有关系只能引用已知节点”的测试。页面状态只组合以下可核对信息，不调用大模型判断完成度：

- `skills/*/config.yaml` 的启用状态。
- `docs/development/TODO.md` 的状态和下一动作。
- 对应代码模块是否存在。
- 写作、审核、运维 Bot 的心跳。

建设成熟度和运行在线状态是两套信息。例如“审核 Bot 已上线·优化中 / 当前在线”表示入口正在正常运行，但质量基线仍在建设；“写作 Bot 已上线·优化中 / 心跳超时”表示能力已经交付，但当前进程需要关注。

截至 2026-07-17，项目总览还必须体现以下已经进入项目的能力和边界：

- 用户入口包括写作 Bot、独立材料润色 Bot、审核 Bot、运维 Bot 和本机控制台；材料润色 Bot 使用自己的心跳，不再混入写作 Bot 状态。
- 写作能力包括直报、单/多素材简报、综合调研、材料润色和深银协动态；综合调研与深银协动态的 Word 结果通过公共附件回传。
- 审核能力包括通用 Word/文字、静态 HTML、公文格式、多文件联合审核和单份 PPTX 低级错误审核；PPT 第一阶段已经上线并处于优化中，但不展示成已具备视觉版式或图片 OCR 审核。
- `TODO-031` 的审核共享核心是待建设能力，只表示规划中的共享问题结构、证据校验、复核和去重，不表示现有审核器已经完成迁移。
- 网页读取与联网搜索节点要体现 DeepSeek 原生搜索形成候选、受限网页读取核验原文的当前链路。

控制台读取四个 Bot 的全部心跳用于本机展示，但不修改运维 Bot 的 `M_AGENT_OPS_MONITORED_SERVICES` 告警配置；“控制台显示某服务”与“运维 Bot 已对该服务启用掉线告警”是两个口径。

关系图使用开源 `vis-network 10.1.0`。压缩脚本和 MIT/Apache-2.0 许可证固定保存在：

```text
app/admin/static/vendor/
```

控制台通过固定只读路由提供该脚本，不开放任意静态目录浏览，也不使用 CDN。选择它是因为当前控制台是 Python 服务端渲染页面，无需为关系图额外引入 React、Node 构建链或联网依赖。

### Skill 开关

后台读取：

```text
skills/<skill_id>/config.yaml
```

点击开启/关闭时，只修改其中的：

```yaml
enabled: true/false
```

说明：

- 开启 skill 只是让它进入路由候选。
- 该 skill 仍然需要有可运行的 `workflow`。
- 用户仍然需要在权限策略里被允许使用该 skill。

### 用户权限

后台读取和写入：

```text
config/platform-policy.yaml
```

用户权限格式：

```yaml
users:
  user-001:
    allowed_skills:
      - direct_report
```

后台页面中多个 skill 用英文逗号分隔：

```text
direct_report, writer1
```

用户权限不会在默认页面读取或展示。管理员需要主动点击“显示用户权限与任务记录”，进入：

```text
/?show_sensitive=1#users
```

保存权限后继续停留在显式展示模式；点击隐藏后返回默认页面。

### 最近任务

后台读取：

```text
../M-Agent-Files/tasks/writing/YYYY/MM/
```

只展示任务摘要，包括：

- job_id
- 用户 ID
- 命中的 skill
- 标题
- 简短消息

不展示 `.env`、API Key 或完整正文。

写作和审核板块会使用任务数量汇总，但不会读取或展示审核原文、段落预览和用户材料。

最近任务和用户权限共用按需敏感区。默认访问 `/` 时，服务端不会调用权限列表和最近任务读取函数，因此数据不只是视觉折叠，也不会出现在默认页面源码中。写作和审核板块中的任务数量汇总仍保留，因为它只统计固定任务目录下的元数据文件数量，不读取任务内容。

## 规划能力：政策库数据管理

目标：让使用者在本机后台里可视化清理底层政策库，减少无关政策进入直报、简报写作材料。

第一阶段只管理本地 SQLite 政策库：

```text
../M-Agent-Files/knowledge/policy/policies.sqlite3
```

后台页面应提供：

1. 政策列表：显示来源、标题、发布日期、分类、链接和入库编号。
2. 筛选搜索：支持按来源、分类、关键词筛选。
3. 勾选删除：支持批量勾选无关政策。
4. 删除预览：删除前展示待删除清单。
5. 自动备份：真正删除前先复制 SQLite 到 `../M-Agent-Files/knowledge/policy/backups/`。

建议数据流：

```text
管理后台页面
  -> app/admin/services.py
  -> app.policy_knowledge.store
  -> ../M-Agent-Files/knowledge/policy/policies.sqlite3
  -> 备份 SQLite
  -> 删除所选 source + doc_id
```

操作原则：

- 以后清理政策库时，优先从后台删除底层 SQLite 数据。
- 后台不提供任意 SQL 输入。
- 后台不提供任意文件浏览。
- 删除必须以 `source + doc_id` 为唯一定位依据。
- 删除必须先备份，再执行。
- 删除结果应显示：删除前数量、删除数量、删除后数量、备份文件路径。

第一阶段不做：

- 不做多人协同后台。
- 不做公网访问。
- 不做复杂权限系统。
- 不做 Obsidian 软件 UI 自动操作。
- 不直接影响企业微信用户侧能力。

## 安全边界

当前后台的安全设计：

- 默认只绑定本机地址。
- 只通过底座配置解析读取数据路径和心跳阈值，不展示或返回 `.env` 内容。
- 不展示密钥。
- 不提供任意文件浏览能力。
- Git 只执行代码内固定的只读查询，不接受页面传入命令，也不自动访问网络。
- 页面当前只能写入 skill 开关和用户权限配置；项目总览、待办、运行状态和更新记录均为只读。
- 用户权限和最近任务默认不加载，只能通过本机页面的显式操作查看。

后续如果要多人使用后台，必须先补：

- 登录鉴权。
- 操作审计。
- CSRF 防护。
- 更细粒度的权限。
- 内网部署方案。

政策库删除能力还必须补：

- 删除前确认页。
- 删除前自动备份。
- 删除后的 Wiki 重建结果展示。
- 删除失败时不覆盖原有 Wiki。

## 后续开发规则

改后台时优先改：

```text
app/admin/
tests/test_admin_*.py
docs/development/admin-console.md
```

不要在后台里直接实现写作、审核等业务能力。业务能力仍然放在：

```text
skills/<skill_id>/
```

后台只负责配置和观察。

## 测试

管理后台相关测试：

```bash
uv run --locked pytest tests/test_admin_services.py tests/test_admin_server.py -v
```

测试至少覆盖：

- TODO 字段解析、开放状态和优先级排序。
- 五层架构完整性，以及 TODO、Skill 开关、代码证据和心跳对功能状态的影响。
- 仓库中每个 `skills/*/config.yaml` 的 id 都映射到至少一个架构能力节点，新增 Skill 未同步总览时测试失败。
- 架构关系只引用已登记能力，关键入口、底座、Skill、知识库和运维关系存在。
- 关系图脚本从本地固定路径加载，版本和许可证文件齐全，页面不依赖 CDN。
- Bot 心跳正常、超时、缺失和损坏文件降级。
- 板块汇总只统计任务和知识库数量，不读取材料正文。
- 页面转义 Skill、TODO 和 Git 摘要，不渲染其中的 HTML。
- 默认页面不调用用户权限和最近任务读取函数，不生成相关导航、表格或正文；显式展示模式仍可正常管理和查看。

政策库数据管理上线前至少覆盖：

- 列表筛选不会返回 `.env`、任务日志或其他本机文件内容。
- 删除操作只按 `source + doc_id` 删除指定政策。
- 删除前会生成备份文件。
- 删除后会触发 Wiki 重建。
- 删除空选择、重复选择、非法 `doc_id` 时返回可理解的错误。

涉及底座联动时再跑：

```bash
uv run --locked pytest tests/test_platform_registry.py tests/test_platform_router.py tests/test_writing_platform_bot.py -v
```
