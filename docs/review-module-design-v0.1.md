# 智能审核模块设计方案 v0.1

> **版本状态**:本文已更新至 v0.2，反映两层审核架构（格式类代码化 + 语义类 LLM CoT）。

> 起草人:leizhou + Hermes 对话对齐
> 起草时间:2026-06-13

## 1. 目标

为 M-Agent 项目新增一个**独立模块**"智能审核",**第一版只审低级错误**,不查风格、不查深度、不改稿。

### 1.1 范围

**做的:**
- 错别字(重复字、形近字)
- 标点错误(中英文混用、标题标点、引号配对)
- 语言通顺(明显搭配错误、重复字/词)
- 格式问题(序号跳号、空行异常、数字/单位规范)
- 审核意见存档

**不做的(第一版明确边界):**
- 多文件类型(只支持 `.docx`)
- 打分 / 总评
- 改稿 / 建议改法
- 政治表述合规检查(放第二版)
- PDF / Markdown / TXT
- 规则热更新(改 `rules.md` 后重启 Bot 才生效)
- 消息卡片 / 富文本(纯文本输出)
- 会话状态(完全被动型)
- 白名单 / 权限控制(复用现有企微 Bot 机制)

## 2. 核心架构

### 2.0 两层审核架构（v0.2 升级）

**设计背景**:LLM 单次调用不稳定，同一文档可能漏报大量问题。

**解决方案**:按规则类型分治 + 多次调用取并集。

#### 格式类规则 → 代码正则检测（稳定）

| 规则 ID | 名称 | 检测方式 |
|---------|------|----------|
| `quote-pair` | 引号不成对 | 状态机追踪开闭引号 |
| `num-unit` | 数字和单位间空格 | 正则 `\d+\.?\d+%?\s+[万千亿个点]?[元万吨百亿个]?` |
| `mixed-punct` | 中英文标点混用 | 正则检测中文句子里英文标点 |
| `toc-no-ordinal` | 目录/章节标题带序号 | 正则 `^[一二三四五六七八九十]+[、.．]` + 关键词匹配 |
| `toc-seq-skip` | 目录序号跳号 | 中文数字转阿拉伯数字后比对 |

#### 语义类规则 → LLM CoT + 结构化输出 + 3次取并集

| 规则 ID | 名称 |
|---------|------|
| `title-truncated` | 新闻标题被截断 |
| `content-mismatch` | 标题和正文不匹配 |
| `content-incomplete` | 正文语义不完整（句中截断） |
| `toc-mismatch` | 目录与正文不匹配 |
| `content-out-of-scope` | 内容不在收录范围 |
| `content-wrong-section` | 内容放错板块 |
| `content-duplicate` | 重复内容 |
| `content-outdated` | 过时信息 |

**LLM 调用策略**:
1. Chain-of-Thought prompt：先识别文档结构，再逐段落分析
2. 结构化 JSON 输出：`{ "reasoning": "...", "issues": [...] }`
3. 3 次调用取并集，去重（rule_id + paragraph_index 相同取第一次结果）
4. 只有 `__llm_error__` 标记的 findings 才触发重试

### 2.1 双端口模型

| 端口 | 形态 | 输入 | 输出 | 谁用 |
|------|------|------|------|------|
| **规则沉淀端** | Hermes 对话 | leizhou 日常丢错文 / 口述 | AI 提炼规则,leizhou 确认后写入 `rules.md` | leizhou |
| **审核执行端** | 企微 Bot | 任何用户丢 `.docx` 文件 | 按 `rules.md` 审,输出审核意见 | 同事/用户 |

### 2.3 关键设计决策

| 决策点 | 方案 | 理由 |
|--------|------|------|
| 入口形式 | **B 双 Bot** | 现有 Bot 做风格沉淀,新申请 Bot 做审核(已实现) |
| 触发方式 | **A 被动型** | 用户丢文件就审,什么话都不用说 |
| ACK 反馈 | 收到文件回"收到,正在审核" | 让用户知道系统在工作 |
| 拒接非文件 | 回固定话术 | "本入口仅接收审核文档(.docx),请直接发送文件" |
| 文档格式 | 第一版只支持 `.docx` | 不贪多,先跑通 |
| 规则库格式 | Markdown(人友好,AI 也能解析) | 比 JSON 直观,方便手工维护 |
| 规则共享 | 共享本地文件 | 同一台机器、Hermes 写、企微 Bot 读 |
| 规则热更新 | **不支持**(重启生效) | 第一版简化 |
| 审核意见格式 | 纯文本 | 错 ≤10 条直接列,>10 条给摘要 + 报告文件 |
| 审核意见存档 | 保存 | 写到 `data/reviews/2026-06-13-XXX/` |
| 改稿建议 | **不做** | 只标错,不教怎么改 |

- `app/review/` 和 `app/main.py` 是**两个独立进程**,各自跑各自的企微 Bot
- `app/main.py` 用现有的 Bot 做风格沉淀(不动)
- `app/review/main.py` 用**新申请的 Bot** 做审核
- 共享规则库文件 `app/data/rules.md`(由 Hermes 写、Bot 读)

## 4. 规则库文件结构

### 4.1 文件路径

`app/data/rules.md`

### 4.2 格式:YAML Frontmatter + Markdown Body

```markdown
# 智能审核规则库

> 最后更新:2026-06-13 18:11
> 维护人:leizhou
> 总规则数:10

---

## 规则 1:重复字检查

**ID:** `dupe-char-check`
**严重程度:** error
**预置规则:** ✅ 是
**创建时间:** 2026-06-13

### 规则描述
检查同一行或相邻行是否存在重复字,如"的的""了了""的的的"。

### 正例
- 我们要加快推进工作

### 反例
- 我们要的的加快推进

### 检查方式
正则匹配: `([\u4e00-\u9fa5])\1{1,}`(连续重复汉字)

---

## 规则 2:中英文标点混用

**ID:** `mixed-punct`
**严重程度:** error
**预置规则:** ✅ 是
**创建时间:** 2026-06-13

### 规则描述
检查是否在中英文混排时,中文上下文里出现了英文标点(逗号、句号、分号、问号、感叹号)。

### 检查方式
- 扫描每行文本
- 如果该行包含中文字符(任意一个汉字),则该行的 `,。;:?!` 必须为中文全角
- 否则报错并指出具体位置
```

### 4.3 结构要点

- 每条规则有**唯一 ID**(英文,程序引用)
- 每条规则有**严重程度**:`error` / `warning`
- 区分**预置规则** vs **leizhou 确认的规则**(都放一起,但有标记)
- **检查方式** 写清楚正则 / 规则细节(给程序看)
- **正反例**给 AI 参考(给 AI 看的,程序可以不用)

## 5. 审核 Bot 对话流程

### 5.1 完整流程图

```
┌──────────────────────────────────────┐
│ 用户丢一个 .docx 文件过来             │
└────────────────┬─────────────────────┘
                 ↓
        ┌─────────────────────┐
        │ Bot 判断消息类型      │
        └─────────┬───────────┘
                 ↓
         ┌───────┴────────┐
         │ 是 .docx 文件?   │
         └──┬───────────┬──┘
          是↓           否↓
   ┌──────────────┐  ┌──────────────────────┐
   │ ACK 提示      │  │ 回固定话术            │
   │"收到,正在审核" │  │"本入口仅接收审核     │
   └──────┬───────┘  │ 文档(.docx),         │
          ↓          │ 请直接发送文件"       │
   ┌─────────────────┐└──────────────────────┘
   │ 解析 .docx       │
   │ 加载 rules.md   │
   │ 逐条规则匹配    │
   └────────┬────────┘
            ↓
   ┌────────────────────┐
   │ 格式化审核意见      │
   └────────┬───────────┘
            ↓
    ┌───────┴────────┐
    │ 错 ≤ 10 条?     │
    └────┬────────┬──┘
      是↓        否↓
  ┌──────────┐ ┌──────────────────┐
  │纯文本    │ │纯文本摘要         │
  │列出全部  │ │+ report.md 文件  │
  └──────────┘ └──────────────────┘
```

### 5.2 审核意见格式

**错 ≤ 10 条(纯文本列出全部):**

```markdown
📄《2026年XX工作汇报.docx》审核完成

错误1:【标题截断】新闻标题被截断,语义不完整
所属段落：「中共中央政治局12月25日召开会议,听取中央纪委国家监委工作汇报,研究部署2026年党风廉政建设和反腐败工作。中共中央总...」

错误2:【内容不完整】正文段在'维护了'后戛然而止,语义不完整,缺宾语
所属段落：「12月25日,公安部在京召开专题新闻发布会,通报公安部和国家金融监督管理总局联合部署开展金融领域'黑灰产'违法犯罪集群打...」
```

**错 > 10 条(纯文本摘要 + 报告文件):**

```markdown
📄《2026年XX工作汇报.docx》审核完成

错误1:【标题截断】...
所属段落：「...」
...
错误10:【数字单位格式】...
所属段落：「...」

📎 完整报告:data/reviews/20260623-001/report.md
```

**输出原则**:
- 不要写"第几段",人在文档里无法数段数
- 直接引用原文片段,方便在文档里搜索定位
- 按规则分组,每条问题单独编号(错误1/错误2...)
- `title-truncated` 的"所属段落"显示**原文**（截断的标题）

### 5.3 拒接非 .docx 消息

固定话术:
```
本入口仅接收审核文档(.docx),请直接发送文件
```

## 6. 规则抽取流程(Hermes 端)

### 6.1 场景 A:丢错文 + 解释

> **leizhou**:这条要入库:"我们要的的加快推进 XXXX"——不能出现重复字
>
> **Hermes**:好,已分析。这段错例可以总结成规则:
>
> **规则候选:**
> - **名称**:重复字检查
> - **ID**: `dupe-char-check`(或你指定的)
> - **严重程度**:error / warning(你定)
> - **检查方式**:正则匹配 `([一-龥])\1{1,}`
> - **正例**:"我们要加快推进"
> - **反例**:"我们要的的加快推进"
>
> **确认方式**:
> 1. 写入 `app/data/rules.md`(我会告诉你写在哪、变成什么样子)
> 2. **等你回"确认"或"入库"** 才真的写
> 3. 你说"不通过"我就不写
> 4. 你说"改成 XXX"我改完再给你看
>
> **leizhou**:确认
>
> **Hermes**:好,已写入。规则总数:9 → 10。要不要现在重启审核 Bot 让它生效?

### 6.2 场景 B:口述一条规则

> **leizhou**:加一条规则,凡是涉及具体数据(百分比、金额、数量)的,后面要注明数据来源
>
> **Hermes**:好。这条规则我会写成:
>
> - **名称**:数据来源标注检查
> - **检查方式**:扫描所有数字 + %/亿/万/元等单位,检查其后 50 字内是否出现"据""根据""来源"等关键词
> - **严重程度**:warning
>
> **确认吗?**(回"确认"我就写)
>
> **leizhou**:确认,但要改成"出现在数据前 50 字也算"
>
> **Hermes**:好,已调整。给你看最终版... 确认?
>
> **leizhou**:入库
>
> **Hermes**:已写入,规则总数:10 → 11。

### 6.3 场景 C:丢错例但不解释

> **leizhou**:这条要入库:"截至 2025 年底,全区共有规上工业企业 386 家,同比增长 12%,产值 1024 亿元,同比增长 15%。(括号内原文如此)"
>
> **Hermes**(自动分析):这看起来是数据引用,我理解你想入库的规则可能是:
>
> - **规则 A**:涉及具体数据时,要核对前后数据是否矛盾(此例 12% 和 15% 是不同维度,不一定矛盾)
> - **规则 B**:涉及数据时,应注明数据来源/出处
> - **规则 C**:其它?
>
> 你想入库哪条?或者都不是,你想入库的是别的?

### 6.4 核心约定

- **AI 提炼建议必须先给 leizhou 确认,不能自动写入 `rules.md`**(与领导风格沉淀规则一致)
- leizhou 说"确认"或"入库"才真的写入
- leizhou 说"不通过"不写
- leizhou 说"改成 XXX"我改完再给他看
- 写入后报告:规则总数变化 + 是否需要重启 Bot

## 10. 风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| `.docx` 解析对复杂排版支持差 | 段号定位可能错 | 第一版假设是纯文本稿件,复杂排版放第二版 |
| 用户丢大文件超时 | Bot 卡住 | 第一版加文件大小限制(比如 10MB) |
| 规则匹配误报多 | 用户体验差 | 第一版每条规则都有"严重程度",先标 error,warning 放第二版启用 |
| 拒接话术被滥用 | 用户困扰 | 话术保持简短,后续可优化 |
