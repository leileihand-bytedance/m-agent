# 材料润色 Skill 设计

> 状态：v1 已实施，边界仍是直接粘贴文字，不支持链接或文件。

## 背景

当前平台已经支持直报、简报和基于上一稿继续改写，但还缺少一个独立的“材料润色”能力。用户希望新增一个新 skill，初期只支持直接粘贴文字，并支持继续对话式修改。

用户明确要求：

- 该 skill 要和直报、简报的改稿严格切开，不能混用。
- 第一版默认输出“润色后正文 + 1-2 句修改说明”。
- 润色风格根据用户要求判断，不预设固定模板。
- 如果用户只说“润色一下”，默认只优化表达，不新增事实、不主动补写新内容。
- 如果原文明显没有写完，例如半句话断开、用了“XX”等占位，可以做最小必要补全；无法确定时应追问。

## 目标

新增独立 `rewrite` skill，用于处理用户直接粘贴的文字润色，并支持在同一会话中继续提出修改要求。

## 非目标

第一版不做以下内容：

- 不读取网页链接
- 不读取 Word/PDF
- 不联网搜索
- 不补充外部事实
- 不和直报/简报的业务规则混写

## 设计结论

### 1. Skill 形态

沿用项目中已经预留的 `rewrite` skill 名称，但第一版能力只开放“材料润色”。

目录结构：

```text
skills/rewrite/
├── SKILL.md
├── __init__.py
├── config.yaml
├── schema.py
├── workflow.py
└── prompts/
    └── draft.md
```

### 2. 输入

第一版只支持两类输入：

1. 用户在消息中直接写明润色要求，并附上原文
2. 基于 `rewrite` 自己的上一版结果继续修改

支持的常见表达：

- `帮我润色这段：……`
- `把下面这段改得更正式一点：……`
- 第一行写要求，空一行后粘贴正文

如果用户只说“润色一下”但没有贴正文，应追问补充。

### 3. 输出

结构化结果包含：

- `title`：第一版固定为空字符串
- `body`：润色后的正文
- `revision_note`：1-2 句修改说明
- `sources`：固定为空列表
- `needs_clarification`
- `message`

企业微信和 CLI 回复时展示：

```text
润色后正文

修改说明：……
```

但会话存储中的正文只保存 `body`，不把 `revision_note` 写进上一稿正文，避免下一轮改稿把说明当成正文一起带入。

### 4. 路由与切割

这是本次实现的第一约束。

`rewrite` 必须和现有直报/简报改稿切开：

- 如果用户是在已有直报/简报上下文里继续说“再正式一点”“再压缩一点”，仍按原 skill 的上一稿修改处理。
- 如果用户消息中明确出现“润色这段/改写下面这段/优化下面文字”等新任务信号，并且同时附带新的正文内容，则即使当前存在活跃直报/简报会话，也要强制识别为新的 `rewrite` 任务。

### 5. 默认规则

当用户只说“润色一下”时，默认遵循：

- 只优化表达、语序、逻辑顺滑度、措辞正式度
- 不新增事实
- 不主动扩写新观点
- 对明显残句或 `XX` 占位，只做最小必要补全
- 无法确定的部分宁可追问，也不臆造

## 验证重点

- `rewrite` skill 可被注册和路由
- 用户直接贴文字可生成润色结果
- 没有正文时会追问
- `rewrite` 自己支持继续对话式修改
- 有活跃直报/简报时，带新正文的润色请求不会误改上一稿
- 回复中能显示修改说明，但会话存储正文不混入修改说明
