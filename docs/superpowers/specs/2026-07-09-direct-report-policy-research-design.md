# 直报政策研究层与 Wiki 下线设计

> 状态：阶段性设计已实施；随后由共享政策研究层扩展并部分取代，不作为当前实现唯一依据。

## 背景

当前直报写作对政策的使用主要依赖通用政策检索和少量摘录，容易出现两个问题：

1. 政策能“勉强挂上”，但不够贴切，像套话。
2. 当政策材料本身不够合适时，模型仍可能被带偏。

同时，现有 Obsidian `wiki` 主要承担人工浏览和整理作用，AI 写作并没有真正使用其中最有价值的“主题页/写作卡片”内容，维护成本和收益不匹配。

## 目标

本次只做两件事：

1. 全项目下线 `wiki` 相关调用、配置、代码、测试和文档。
2. 为 `direct_report` 增加第一版“政策研究层”，先只深做 `小微金融`、`科技创新` 两个主题。

## 非目标

本次不做：

1. 不重做 `writer1` / `writer2` 的政策逻辑，只移除 `wiki` 依赖。
2. 不做自动联网补政策并直接写稿；如果本地没有贴切政策，直报默认直入主题。
3. 不尝试一次覆盖所有政策主题。

## 方案选择

### 方案 A：继续增强通用政策检索

优点：改动小。

缺点：本质仍是“搜到一条就塞给模型”，很难把政策逻辑写深。

### 方案 B：下线 Wiki，新增直报专用政策研究层

优点：可以先判断“需不需要挂政策、该挂哪一层、怎么转到微众银行”，更符合直报写作需求。

缺点：实现成本更高，需要新规则和测试。

### 结论

采用方案 B。

## 设计

### 1. Wiki 下线范围

下线范围包括：

1. `policy_wiki_materials` 工具入口。
2. `app.policy_knowledge.wiki` 代码和 CLI 导出命令。
3. skill 配置中的 `policy_wiki_materials` 授权。
4. 平台配置中的 `policy_wiki_vault_dir`。
5. 相关自动化测试和文档。

### 2. 直报政策研究层定位

政策研究层只服务 `direct_report`，位于 skill 内部，不放进平台底座公共层。

职责不是“返回一批政策材料”，而是先产出写作研究结论：

1. 当前素材是否适合挂政策。
2. 是否命中受支持主题。
3. 建议使用哪一条政策。
4. 这条政策为什么合适。
5. 开头怎么写更自然。
6. 如何从政策转到微众银行动作。
7. 结尾如何抬升。

### 3. 第一版支持主题

只支持：

1. `小微金融`
2. `科技创新`

其它主题第一版不强挂政策，默认直入主题。

### 4. 候选政策选择原则

先按第一性原理定义“直报真正需要的政策类型”，再到存量政策库里找，而不是默认相信库存量。

#### 小微金融

可用政策应能自然支撑以下逻辑：

1. 服务实体经济、普惠金融、小微企业融资支持。
2. 提升融资可得性、信用贷款、首贷续贷、数字化金融服务。
3. 能解释微众银行相关举措为什么具有政策意义。

#### 科技创新

可用政策应能自然支撑以下逻辑：

1. 科技创新、科技金融、服务创新主体。
2. 支持科创企业、创新链产业链、新质生产力等方向。
3. 能解释微众银行相关做法如何服务创新发展。

### 5. 明确排除规则

以下材料即使命中关键词，也不作为优先政策：

1. 会见、致辞、泛部署、弱相关会议稿。
2. 只能显得正式、但不能支撑正文逻辑的政策。
3. 不能自然转到微众银行行动的政策。
4. 只沾了关键词、但与素材主线不贴合的政策。

### 6. 写作链路变化

`direct_report` 改为：

1. 读取用户素材。
2. 判断是否适合挂政策。
3. 仅当素材属于 `小微金融` 或 `科技创新`，再进入政策研究层。
4. 从本地政策库中按严规则挑选一条最贴切政策。
5. 若没有贴切政策，则直入主题，不追加政策材料。
6. 把政策研究结论写入 `planning_note`，而不是只给模型一段通用摘录。

## 测试策略

需要新增或调整的测试包括：

1. 平台工具不再暴露 `policy_wiki_materials`。
2. CLI / 配置不再包含 `policy_wiki_vault_dir`。
3. `writer1` / `writer2` 不再尝试读取 `wiki`。
4. `direct_report` 在小微金融素材中可选到贴切政策。
5. `direct_report` 在科技创新素材中可选到贴切政策。
6. 候选政策不贴切时，不追加政策，直接入题。
7. 非支持主题不强挂政策。

## 验收标准

1. 仓库中不再残留 `wiki` 写作链路、导出命令和相关配置。
2. 直报在 `小微金融`、`科技创新` 两类稿件中，政策背景明显更贴题。
3. 本地无合适政策时，系统稳定直入主题，不再硬挂一层空泛政策。
4. 自动化测试覆盖以上关键行为。
