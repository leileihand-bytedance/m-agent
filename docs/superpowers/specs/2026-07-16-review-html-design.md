# HTML 文字审核设计

## 目标

在现有独立审核 Bot 中增加单个 `.html` / `.htm` 文件审核能力。系统只审核页面静态正常显示的文字，检查错别字、名称、语病、标点、内容残缺、重复内容，以及同一页面内日期、数量、金额、比例、状态和统计口径等数据表述是否前后一致。审核完成后只返回审核消息，不生成或返回带标记文件。

## 范围与边界

- 只支持用户上传的单个 `.html` 或 `.htm` 文件，不把 HTML 加入现有 `.docx` 多文件联合审核。
- 不执行 JavaScript，不下载外部脚本、样式、接口数据、图片或字体，不访问 HTML 中的链接。
- 忽略注释、`head`、`script`、`style`、`template`、`noscript`、`svg`、`canvas` 等非正文内容。
- 忽略带 `hidden`、`aria-hidden="true"`，或行内样式明确包含 `display:none`、`visibility:hidden` 的元素及其子元素；关闭的 `dialog` 不提取，折叠的 `details` 只提取首个可见 `summary`。
- 第一版不解释复杂 CSS 选择器。仅通过页面内 CSS 类隐藏、但没有显式隐藏属性或行内隐藏样式的文字，可能仍被提取。
- 只读取文本节点；不把标签属性、图片替代文字、链接地址、表单值或元数据当作正文。
- 不联网核查外部事实，只判断当前 HTML 内部的文字质量和前后自洽性。

## 技术方案

### 1. HTML 静态文字解析

在 `app/review/` 新增独立 HTML 解析器，使用 Python 标准库 `html.parser.HTMLParser`：

- 按 DOM 输入顺序处理静态文本，不运行页面代码。
- 以标题、段落、列表项、表格行及常见块级容器作为段落边界。
- 表格按行输出，每一行的单元格用稳定分隔符连接，保留表头、指标名称、统计时间和数值之间的邻接关系，供数据一致性审核使用。
- 合并连续空白，过滤空段落，并保留可定位的段落顺序。
- 解码优先级为 UTF BOM、真实 HTML `meta charset`、UTF-8、GB18030；注释、脚本和正文中的伪 `charset=` 不参与编码判断，无法可靠解码时返回可理解的文件解析失败提示。
- 对畸形但标准库仍可容错读取的 HTML 尽量提取；完全无可见文字时明确拒绝审核。

解析器只接收已冻结在当前审核任务 `input/` 下的文件，不能读取任务目录外路径。

### 2. 网页 PPT 页码定位

HTML 没有统一的物理页概念，浏览器打印分页还会随窗口、字体和打印设置变化。对本项目实际网页 PPT，页码按源码中明确的页面容器确定：

- 元素的 class 列表精确包含 `slide` 时，视为一页；按 DOM 出现顺序从第 1 页连续编号。
- 页面容器内提取的标题、段落、列表项和表格行都记录所属页码；跨页数据矛盾定位到模型返回的待修改段落所在页。
- 页面内的 `page-label` 等展示文字不作为页码来源，避免封面、章节页未显示页脚时造成页码偏移。
- 没有 `slide` 容器的普通 HTML 不伪造页码，审核消息改用“第 N 段”定位，同时保留所属段落原文供搜索。
- 不执行 JavaScript，也不启动浏览器计算 CSS 打印分页。

解析结果在段落列表之外增加一份等长的页码映射。通用审核仍只接收原段落列表；页码映射仅在格式化 HTML 审核消息和归档报告时使用，因此 Word、直接文字和其他审核类型的输出完全不变。

### 3. 审核引擎复用

HTML 提取出的段落直接复用 `app/review/general_reviewer.py` 的通用审核能力和 `app/review/rules_general.md` 规则，不复制一套 HTML 专用文字规则。

现有通篇逻辑校对默认只覆盖总文本 `200` 至 `100000` 字。为满足 HTML 数据一致性要求，通用审核入口增加一个显式、默认关闭的参数，允许 HTML 在至少存在两个正文段落时把通篇逻辑校对下限降为 `0`。`.docx` 和直接文字保持原行为不变。

数据一致性重点包括：

- 同一指标、对象、时间和口径下的数量、金额、比例或状态前后矛盾。
- 汇总数与分项数明显无法同时成立。
- 标题、摘要、正文或表格对同一数据的表述冲突。
- 起止日期、时间顺序或统计范围自相矛盾。

仍沿用现有保守原则：时间、对象或统计口径不同且无法证明冲突时不报，避免把累计值与当期值等合理差异误判为错误。

### 4. 入口、任务和结果

- 审核入口接受 `.html` 和 `.htm`，原 `.docx`、直接文字、公文格式审核和多文件联合审核行为不变。
- HTML 作为新的单项任务类型进入审核专用 SQLite 持久队列，复用消息幂等、排队、租约、恢复、运维告警和主动发送能力。
- 原始 HTML 在入队前冻结到当前任务 `input/`，任务载荷只保存安全相对路径，不保存 HTML 正文。
- 后台解析并审核后，把审核报告写入 `output/report.md`，状态索引不写用户正文。
- 企业微信只主动返回 Markdown 审核消息；网页 PPT 有问题时列出“位置：第 N 页”、原文和说明，普通 HTML 列出“位置：第 N 段”，无问题时明确告知未发现明显问题。
- `output/report.md` 使用同一套 HTML 位置映射，保证用户消息与归档报告一致。
- HTML 不调用 Word 标注逻辑，`output/` 不生成 `marked_*.html` 或其他标注文档。

### 5. 错误处理与安全

- 后缀不支持时继续返回现有格式提示，但提示范围更新为 `.docx`、`.html/.htm` 或直接文字。
- 空文件、无可见文字、无法解码、解析失败和超出既有上传限制分别返回简短、可操作的提示，详细异常只进入安全日志和运维事件。
- HTML 内容一律视为不可信输入；标签名、属性和文本都不能影响文件路径、任务类型、系统提示或工具权限。
- 不增加浏览器、JavaScript 运行时或网络访问能力，不新增第三方依赖。

## 测试设计

按测试先行覆盖：

1. 提取标题、段落、列表和表格可见文字，并保持稳定顺序。
2. 忽略脚本、样式、注释、模板、隐藏属性、`aria-hidden` 和行内隐藏样式。
3. 不读取标签属性、链接地址和图片替代文字。
4. 支持 UTF-8、BOM、`meta charset` 和 GB18030，空正文或无法解码时安全失败。
5. `.html/.htm` 被入口接受，其他旧格式行为不变；HTML 不进入 `.docx` 多文件组装。
6. HTML 任务冻结输入、载荷白名单、目录隔离、幂等恢复和 Markdown 主动发送。
7. HTML 审核复用现有通用规则；短 HTML 也触发全文逻辑检查，能识别同口径数据矛盾，并放过不同时间或累计/当期口径。
8. 只生成 `report.md` 和审核消息，不生成标记文件。
9. 网页 PPT 的 `slide` 容器按 DOM 顺序映射页码，标题、正文和表格错误显示正确页码；跨页矛盾定位到待修改段落所在页。
10. 普通 HTML 没有 `slide` 时显示段落位置，Word 和直接文字消息保持原样。
11. 更新审核模块、底座、能力、开发路线和测试交付文档，并通过核心文档检查。

## 方案比较与选择

1. **按明确 `slide` 容器映射（采用）**：与实际网页 PPT 结构一致，结果稳定，不需要执行脚本；普通 HTML 可安全回退到段落位置。
2. **启动浏览器按打印布局分页（不采用）**：页码受窗口、字体、打印参数和动态资源影响，也会扩大当前“不执行页面”的安全边界。
3. **把所有 `section` 或块级容器当页面（不采用）**：普通网页中的章节和布局容器不等于页面，容易产生大量错误页码。

## 不在本次范围

- JavaScript 动态渲染文字。
- 外部资源加载、链接抓取或网页在线审核。
- CSS 选择器和完整浏览器布局计算。
- 浏览器打印页码、仅由 CSS 分页产生的页码，以及没有明确 `slide` 容器时推测物理页。
- 图片 OCR、图表视觉识别、HTML 样式或页面排版审核。
- 多个 HTML 或 HTML 与 Word 的联合审核。
- 在 HTML 源文件中插入批注、高亮或修改建议。
