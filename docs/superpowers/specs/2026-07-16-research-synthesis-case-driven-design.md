# 综合调研案例驱动改造设计

## 背景

现有 `research_synthesis` 已采用“材料台账—正文生成”两阶段，但默认把提纲视为必须完整保留的固定骨架。三组经授权案例证明，综合调研至少存在两种不同任务：

- 问卷式提纲：每个问题都必须回应，缺少素材时要明确标注。
- 政策清单式提纲：提纲是选择范围，只保留与本单位相关且有素材支撑的事项。

案例还表明，终稿可能使用案例包外的补充材料或图片中的数据。因此，终稿可以用于学习选材和组织方式，不能反向证明模型可以补写来源不明的信息。

## 目标

把综合调研工作流升级为“提纲解释—证据台账—正文编排”的逻辑三阶段，并保持两次模型调用：第一阶段同时完成提纲解释和证据台账，第二阶段只根据已核验台账写正文。

改造后应满足：

1. 区分必答型和选择型提纲。
2. 必答型保留全部必答主题及顺序；选择型只保留有明确关联和证据的主题。
3. 每个正文事实都能追溯到本次材料或经核定的派生计算；人工待核验候选只进入台账，不作为已确认事实进入正文。
4. 台账之外的数字不能静默进入正文。
5. 保持当前部门来源标签、图片提醒、Word 输出、首尾备注和文件安全边界。

## 非目标

本轮不做：

- 不联网补充事实或政策。
- 不把真实案例文件、原文、业务数据或文件名提交到 Git。
- 不自动 OCR 或理解 DOCX 内嵌图片；只为未来的图片候选事实预留结构，当前继续输出人工评估提醒。
- 不新增宣传化润色、报告开头结尾自动写作或成稿审核。
- 不解决超长材料逐块穷尽，也不把 `research_synthesis` 切入持久化任务队列。

## 方案比较

### 方案 A：只修改提示词

优点是改动小。缺点是模型即使正确判断了提纲类型，现有确定性校正仍会强制补回原提纲一级标题，也无法阻止台账外数字进入正文。

### 方案 B：新增独立的提纲分类模型调用

优点是阶段边界最清楚。缺点是每个任务增加一次模型调用，延迟、成本和失败面都会增加，而且分类结果仍需与材料台账联合校验。

### 方案 C：扩展现有台账阶段，并增加确定性校验

第一阶段一次返回提纲类型、覆盖模式、选题结果和证据台账；第二阶段只消费台账。代码根据覆盖模式执行不同的标题补全和数字追溯规则。

采用方案 C。它能解决案例暴露的核心问题，同时不增加模型调用次数。

## 数据结构

### 提纲解释

`ResearchSynthesisPlan` 增加：

- `outline_type`：`questionnaire`、`policy_catalog`、`report_skeleton` 或 `unknown`。
- `coverage_mode`：`exhaustive` 或 `selective`。
- `classification_reason`：简短说明判断依据。
- `required_headings`：必答型必须保留的主题。
- `selected_headings`：本次正文实际采用的主题及顺序。
- `omitted_outline_items`：选择型中未采用的提纲事项及理由，只进入台账，不进入正文。

`unknown` 不能静默回退为任意一种模式。模型无法可靠判断且用户说明也不足时，返回一次简短澄清。

### 证据点

`ResearchEvidencePoint` 增加：

- `evidence_kind`：`source_text`、`derived`、`image_candidate` 或 `external_missing`。
- `source_locations`：可用时记录页码、幻灯片号、表格或正文位置。
- `time_scope`、`metric_scope`、`unit`：用于判断数据能否合并。
- `verification_note`：记录待核验原因。
- `usable`：是否允许进入正文。

使用规则：

- `source_text`：有本次材料来源，可以进入正文。
- `derived`：必须记录原始数值、运算和结果，且时间、对象、单位、范围一致。
- `image_candidate`：当前只作为人工核验候选，`usable=false`，不得直接进入正文。
- `external_missing`：素材或用户说明提到另有补充数据、但本次文件中无法追溯的信息，`usable=false`，不得进入正文。

## 工作流

### 1. 识别唯一提纲

沿用现有“用户点名优先—明确文件名—正文答复特征—无法判断则追问”规则，不按上传顺序猜测。

### 2. 提纲解释和证据台账

第一阶段模型同时完成：

1. 判断提纲类型和覆盖模式。
2. 拆分提纲中的主题、责任部门注释和题内子问题。
3. 将部门事实按提纲问题归并。
4. 记录冲突、缺口、图片提醒、派生计算和不允许使用的信息。

确定性校验随后检查：

- 必答型的 `required_headings` 是否覆盖原提纲主题。
- 选择型的每个 `selected_heading` 是否至少有一个可用证据点。
- 派生数值是否写明原值、运算和结果。
- `image_candidate`、`external_missing` 是否被错误标为可用。

### 3. 正文编排

第二阶段只根据通过校验的台账写正文：

- 必答型按 `required_headings` 保留顺序，缺项使用“【材料待补充】”。
- 选择型按 `selected_headings` 组织，不补回未选中的政策大类或事项。
- 主题较稀疏时写成短段；事实、措施或建议较密集时增加二级、三级结构。
- 每个事实段末保留一次规范化来源标签，供当前核对稿回溯。
- 不使用 `usable=false` 的证据点。

### 4. 确定性成稿校验

标题补全改为按覆盖模式执行：

- `exhaustive`：以必答主题为准补全。
- `selective`：只以已选主题为准，不再按原始提纲全部补回。

标题规范化按语义角色处理：

- 与预期一级主题匹配的编号转为“一、”。
- 明确的二级标题转为“（一）”。
- 三级措施项可以保留 `1.`，不再把所有未匹配的阿拉伯编号机械改成“一是”。

事实段必须带有可匹配到本次材料的来源标签；没有合法来源标签时标记“【来源：待核对】”。数字追溯采用保守规则：正文事实段出现的数字必须能在可用证据点或经核定派生结果中找到。不能追溯时，不让该数字作为已确认事实通过，改为“【来源待核对】”提示。标题、层级编号、政策文号和用户提纲标题中的年份不按事实段数字处理。

## 图片处理

当前统一文档服务会提取 DOCX 图片并保留原位置提醒，但不会识别图片内容，也不会把图片资产内容交给本 skill。因此本轮继续：

- 不把源图片嵌入汇总稿。
- 在相关小节保留部门和数量提醒。
- 在 schema 中预留 `image_candidate`，但当前不得自动标为可用。

后续若单独建设“DOCX 图片按需 OCR”，OCR 结果必须带图片位置、识别置信度和人工确认状态，再填入 `image_candidate`；本次正文规则无需随之放宽。

## 案例基线与测试

真实案例只用于人工研究，不进入仓库。仓库新增三个脱敏等价场景：

1. 政策清单型：原提纲包含多个政策大类，素材只支撑其中若干项；结果不得补回无关大类。
2. 问卷必答型：提纲含连续问题，素材缺少其中一项；结果必须保留顺序并标注缺口。
3. 混合证据型：多部门同口径数字可以形成留痕合计，同时存在图片提醒和一个台账外数字；合计可用，图片事实和台账外数字不得直接入稿。

测试先验证当前实现失败，再实施最小改动。专项测试至少覆盖：

- 提纲类型和覆盖模式进入第一阶段结构化输出。
- 选择型不强制补齐原提纲所有一级标题。
- 必答型继续补齐遗漏主题。
- `derived` 数值具备完整计算留痕时可进入正文。
- 台账外数字被标记为来源待核对。
- 图片提醒数量和原位置语义保持不变。
- 旧的提纲识别、来源规范化、Word 输出和安全测试继续通过。

不使用真实业务数据断言，也不要求离线测试调用真实模型。

## 验收标准

1. 脱敏政策清单场景不会被强制补回所有原始政策大类。
2. 脱敏问卷场景仍完整覆盖全部必答问题和顺序。
3. 台账之外的数字不会作为无提示的已确认事实进入正文。
4. 安全合计具有原值、口径和算式记录；口径不一致时不合计。
5. 图片继续不嵌入成稿，且提醒不丢失；图片候选事实在未确认前不进入正文。
6. 现有综合调研专项、平台运行、文档服务和写作入口离线回归通过。
7. 核心文档检查通过，真实案例材料、日志、本机路径和用户数据未进入提交。
