# 深银协动态选稿与摘编优化设计

> 状态：已确认，进入实施

## 背景与问题

深银协动态用于向深圳市银行业协会报送微众银行的正面新闻和成果，不是银行业信息综合或同业横向盘点。

2026 年 7 月 16 日真实任务错误入选《民营银行也分红，微众等已连续两年派现，留存利润补充资本仍是主流》。该稿虽然标题含“微众”，但正文主要比较多家民营银行。现有规则只要标题含“微众”，就把候选判为核心主体，未检查微众银行内容占比、独立成果段落和报送价值，也未调用已经预留的模型相关性判断。

## 目标

1. 把“微众银行正面新闻和成果报送”设为选稿第一原则。
2. 由代码负责日期、来源、正文可读性、链接、去重和原文校验；由模型负责核心主体、正面成果和摘编可用性判断。
3. 优先采用微众银行专题报道全文；不足时扩大可信外部信源；仍不足时才从综合报道中摘取微众银行相关内容。
4. 综合稿只做抽取式摘编，不全面改写正文，不补充原报道没有的事实。
5. 保留原文链接，摘编稿在文末明确说明来源和摘编性质。

## 不变边界

- 发布日期仍以原文页面标注日期为准。
- 微众银行官网、公众号等自有渠道不作为入选信源。
- 自媒体、营销号、内容农场、无正文聚合页和主体不明转载页不进入候选池。
- 合格数量不足时允许输出 1 篇；0 篇时明确告知，不用低质量材料凑数。
- 搜索摘要不能作为正文，最终内容必须来自 `web_reader` 读取的原文。

## 方案

采用“代码硬条件 + 模型业务判断 + 抽取结果校验”的分层方案。

### 第一层：现有权威媒体专题稿

在现有白名单内搜索。候选通过代码硬条件后，由模型判断是否满足：

- 微众银行是唯一主体或明显主角；
- 主要内容是微众银行的经营成果、普惠金融成果、科技成果、社会价值、合作成果、外部认可或其他可报送正面动态；
- 全文不是同业盘点、行业名单或多机构并列综述。

符合时保留原文全文和原标题。

### 第二层：扩大可信外部信源

第一层不足 3 篇时，再搜索扩展白名单。扩展范围限于：

- 可信全国性行业媒体；
- 可信广东、深圳地方主流媒体；
- 政府、监管、行业协会等权威公开平台。

扩展来源仍执行相同日期、正文、主体和正面成果判断，不因来源扩大而降低内容标准。符合专题稿标准时保留原文全文和原标题。

### 第三层：综合报道抽取式摘编

前两层仍不足 3 篇时，允许使用可信媒体的综合报道，但必须同时满足：

- 原文包含完整、正面、可独立成立的微众银行成果信息；
- 相关内容不是名单式一句话提及；
- 删除其他银行内容后，剩余段落仍能让读者理解事件、成果和必要数据；
- 模型能返回原文中连续、可逐字核验的相关段落。

综合稿处理规则：

1. 正文只由原文中与微众银行直接相关的段落组成，保持原句，不做全面改写。
2. 允许生成突出微众银行成果的新标题，但标题不得增加原文没有的事实、评价、数据或因果关系。
3. 文末与原文链接一起标注：`说明：本文根据原报道中微众银行相关内容摘编。`
4. 模型返回的每个摘编段落必须能在规范化原文中逐字找到；找不到、顺序错乱或内容过短时拒绝该候选。

## 模型结构化判断

为单篇候选增加结构化评估结果：

- `decision`：`full_text`、`extract` 或 `reject`；
- `is_positive_achievement`：是否属于可报送的正面新闻或成果；
- `subject_strength`：`primary`、`substantial` 或 `mention`；
- `reason`：内部筛选理由；
- `suggested_title`：仅 `extract` 时允许填写；
- `excerpt_paragraphs`：仅 `extract` 时返回原文逐字段落；
- `achievement_types`：成果类型标签，用于内部审计和题材去重。

判断约束：

- `full_text` 只允许 `subject_strength=primary` 且 `is_positive_achievement=true`。
- `extract` 只允许 `subject_strength=substantial` 且 `is_positive_achievement=true`。
- `mention`、负面风险事件、中性行业盘点或无法独立成篇的内容一律 `reject`。
- 模型判断不能覆盖代码已经判定的日期、域名、正文不可读或链接不安全失败。

## 数据流

```text
计算半月日期范围
  -> 第一层搜索与网页核验
  -> 模型判断专题全文 / 可摘编 / 拒绝
  -> 专题全文不足 3 篇时执行第二层扩展来源搜索
  -> 再次网页核验和模型判断
  -> 仍不足 3 篇时，从两层可信候选中启用可摘编项
  -> 代码逐字校验摘编段落和标题边界
  -> 同稿、同事件和题材去重
  -> 选取 1—3 篇
  -> 生成 Word 和原文链接
```

专题全文始终优先于摘编稿；媒体权威性、新闻价值和题材差异只在同一内容模式内参与排序。

## 结果与审计

`SelectedArticle` 增加：

- `content_mode`：`full_text` 或 `extract`；
- `source_title`：原报道标题；
- `editor_note`：摘编说明，全文稿为空。

任务内部候选记录保留媒体、发布日期、模型决定、理由、摘编校验结果和淘汰原因。成品 Word 不写内部评分，只展示稿件内容、媒体、发布日期、原文链接；摘编稿额外展示文末说明。

## 异常与降级

- 模型判断失败：该候选不入选，继续处理其他候选；全部候选均无法判断时返回安全失败提示。
- 模型返回自由改写而非原文段落：逐字校验失败并拒绝，不尝试猜测修复。
- 扩展搜索仍无合格稿件：按实际数量输出，0 篇时不生成 Word。
- Word 生成失败：保留结构化结果和明确错误，不返回路径不明的文件。

## 测试要求

1. 用昨天的综合民营银行报道制作脱敏等价回归：标题含“微众”不能再被规则直接认定为全文专题稿。
2. 模型判为 `primary + positive` 时保留全文。
3. 模型判为 `substantial + positive` 时只接受原文逐字段落，并写入摘编说明。
4. 摘编段落不存在、顺序错误、只剩一句名单提及时拒绝。
5. 负面新闻、中性行业盘点和其他银行为主的报道拒绝。
6. 第一层不足时才执行扩展来源搜索；专题全文优先于摘编稿。
7. Word 中全文稿不加摘编说明，摘编稿在原文链接处显示说明和原始标题。
8. 日期、白名单、去重、0/1/2/3 篇降级和安全边界原有测试继续通过。
