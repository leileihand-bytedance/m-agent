# PPT 低级错误审核第一阶段设计

## 目标

在现有独立审核 Bot 中增加单个 `.pptx` 文件审核。第一阶段只检查可编辑内容中的低级错误，并通过企业微信文字消息返回问题清单，不修改原 PPT，不生成 Word、Excel 或其他报告附件。

第一阶段重点发现：

- 错别字、语法语病、标点，以及同一 PPT 内有明确标准写法证据的名称错写或不一致。
- 标题、列表等同组序号跳号、重复或明显倒序。
- 同一 PPT 内同一对象、时间和统计口径下的名称、日期、金额、数量、比例、状态或汇总数据前后不一致。

第二阶段再单独设计多个 PPT 之间的一致性审核，不把跨 PPT 能力混入本阶段。

## 已确认的产品边界

- 一次只审核一个 `.pptx` 文件。
- 审核可编辑文本框、表格文字和数据、图表标题、系列名称及底层可读取数据。
- 不审核演讲者备注。
- 不对截图、扫描图片或其他嵌入图片做 OCR；结果中明确说明图片文字未纳入审核。
- 只检查 PPT 内部一致性，不联网核实外部事实或业务数据真实性。
- 不因不同年份、统计对象、时间范围、单位或“累计/当期”等口径差异直接判错。
- 结果只列事实问题和定位证据，不给修改建议。
- 旧版 `.ppt` 不在范围内，用户需先另存为 `.pptx`。

## 独立性要求

PPT 审核是独立审核项目，业务代码放在：

```text
app/review/ppt/
```

该目录独立拥有：

- 审核输入、问题和结果数据结构。
- 确定性规则及其测试。
- 模型提示词和结构化输出定义。
- 原文证据校验、合并和去重逻辑。
- 企业微信文字结果格式化逻辑。

PPT 审核运行时不得导入或调用：

- `app/review/general_reviewer.py`
- 文字审核流程
- Word 审核流程
- 内参、半月报或公文格式审核引擎
- 通用审核规则加载器

引号、标点、错别字、语病等规则如果与现有审核相同，可以复制规则内容或必要代码到 `app/review/ppt/`，再按 PPT 的“页码—对象—原文”结构独立维护。其他审核项目以后调整、重构或关闭，不得改变 PPT 审核行为。

PPT 审核只复用下列底座公共能力：

- `.pptx` 文件安全校验和统一文档解析。
- 任务目录、输入冻结和文件隔离。
- 持久任务队列、消息幂等、排队、重启恢复和安全状态记录。
- 模型连接配置和受控模型调用基础能力。
- 企业微信主动文字消息发送和运维告警。

## 模块划分

建议按以下职责拆分：

```text
app/review/ppt/
├── __init__.py
├── models.py          # 页、对象、问题、跨页证据和最终结果
├── extractor.py       # 把底座解析结果转成 PPT 审核结构
├── rules.py           # 序号、标点、占位符等确定性规则
├── reviewer.py        # 分页文字审核和全篇一致性审核编排
├── evidence.py        # 页码、对象和原文证据核验、去重
├── formatter.py       # 企业微信文字结果和安全拆分
└── prompts/
    ├── language.md    # 错别字、语病、名称和标点审核
    └── consistency.md # 同一 PPT 内跨页一致性审核
```

每个模块只有一个明确职责。入口不写 PPT 业务规则，底座解析器不写审核判断，模型结果不能绕过证据校验直接发给用户。

## 数据结构

PPT 解析后转为独立审核结构：

```text
PptReviewDocument
  -> slides[]
     -> slide_number
     -> elements[]
        -> element_id
        -> element_type: text | table | chart
        -> text
        -> paragraphs/cells/series
        -> bbox
        -> style（仅保留第一阶段判断所需字段）
```

第一阶段的问题结构至少包含：

```text
PptFinding
  -> rule_id
  -> category
  -> slide_number
  -> element_id
  -> target_text
  -> related_slide_number（仅跨页问题）
  -> related_element_id（仅跨页问题）
  -> related_text（仅跨页问题）
  -> description
  -> evidence_status
```

结果结构不包含 `suggestion` 字段，避免模型或格式化层生成修改建议。

## 审核流程

### 1. 接收与建任务

审核 Bot 接收 `.pptx` 后立即确认收件，冻结文件到当前审核任务 `input/`，登记 PPT 专用持久任务。重复企业微信消息使用幂等键合并，不重复建立任务。

`.docx`、纯文字、内参、半月报和公文格式审核继续走原流程。PPT 入口和执行失败不得影响其他审核类型。

### 2. 安全解析与结构整理

通过底座统一文档服务校验并解析 `.pptx`。PPT 独立 `extractor.py` 只消费底座返回的页面、对象、文本、表格、图表和位置数据，并补成适合审核的页内层级。

如果现有底座解析结果缺少序号审核所需的段落、项目符号层级或表格单元格信息，只在底座 PPT 解析器中补充通用结构字段，不把任何审核规则写入底座。

备注和图片资产不进入审核文本。图表只使用底层可读取标题、系列名称和数据，不根据渲染图片猜测内容。

### 3. 确定性规则

独立代码规则优先检查能够稳定确认的问题：

- 引号不成对、连续标点和中英文标点明显混用。
- 未清理的明确占位符。
- 同一列表或同一标题层级内序号跳号、重复和明显倒序。
- 表格或图表中能够明确识别“总计—分项”关系时的确定性加总矛盾。

序号检查必须先按页面、对象和项目符号层级分组，不能把不同文本框、不同页面或小数编号拼成一条连续序列。

无法确定“总计—分项”关系时不做数学推断，交由全篇一致性审核或直接放过。

### 4. 分页语言审核

模型按页面或相邻小批次审核错别字、语法语病、名称错写或内部不一致，以及标点问题。名称问题只能依据同一 PPT 中的标准写法证据或 PPT 模块自己维护的明确术语规则，不能联网猜测。模型必须返回页码、对象编号、错误原文和事实问题说明，不得返回修改建议。

模型输出只作为候选。代码必须确认：

- 页码存在。
- 对象属于该页。
- `target_text` 能在该对象原文中精确找到。
- 问题说明没有把正确写法虚构成原文。

任一项不满足就丢弃候选。

### 5. 全 PPT 内部一致性审核

先从整份 PPT 建立名称、日期、金额、数量、比例、单位、时间范围和状态索引，再由模型只判断需要跨页阅读才能确认的矛盾候选。

每条跨页问题必须同时提供两侧页码、对象编号和真实原文。证据层至少确认：

- 两侧原文均能在指定对象中找到。
- 两侧讨论的是同一对象。
- 时间范围和统计口径一致或原文没有显示差异。
- 单位可以直接比较，或存在明确、可验证的换算关系。
- 不是“累计/当期”“目标/实际”“全行/某地区”等合理口径差异。

无法证明同口径时不报告。第一阶段不联网判断哪一个数字正确，只指出两处内部表述不一致。

### 6. 合并与文字交付

确定性问题、语言问题和跨页问题统一按“页码—错误原文—问题类别”去重。同一原文同时被多个近似类别命中时，只保留最具体的一条事实问题。

普通问题格式：

```text
1.【第3页｜语病】
原文：……
问题：……
```

跨页问题格式：

```text
2.【第4页 ↔ 第12页｜数据不一致】
原文一：……
原文二：……
问题：两处描述对象、时间和口径相同，但数据不同。
```

结果不包含“建议”“修改为”或替用户选择正确口径的内容。

问题较多时按企业微信主动 Markdown 消息安全长度拆分，多条消息保留连续编号。没有发现问题时直接返回未发现问题，并说明图片文字和备注未纳入本次审核。后台任务编号正常不向用户展示，只有异常需要人工介入时显示处理编号。

## 错误处理与降级

- 文件扩展名不是 `.pptx`：沿用其他审核类型分流；旧 `.ppt` 明确提示另存为 `.pptx`。
- 文件损坏、加密、伪造或安全校验失败：停止处理并返回安全提示，不调用模型。
- 没有可审核文本：说明没有读取到可编辑文字，不返回“审核通过”。
- 个别对象读取失败：记录结构化告警；如果无法证明审核范围完整，结果必须列明未成功读取的页或对象。
- 分页语言审核失败：任务标记为未完整完成，不把缺失阶段伪装为零问题。
- 全篇一致性审核失败：可以返回已完成的确定性和分页结果，但必须明确说明“一致性检查未完成”。
- 企业微信发送状态无法确认：停止自动重发并告警，避免用户收到重复结果。
- Bot 或 worker 重启：复用处理检查点；已完成模型阶段不重复调用，已确认发送成功的结果不重复发送。

## 第一阶段明确不做

- 不修改、批注或重新生成原 PPT。
- 不生成 Word、Excel、PDF 或其他附件报告。
- 不提供修改建议。
- 不审核字体、颜色、对齐、留白、遮挡、溢出或母版一致性。
- 不审核图片中的文字。
- 不审核演讲者备注。
- 不联网查证事实。
- 不审核多个 PPT 之间的一致性。

## 测试设计

所有实现按测试先行推进。

### 独立模块测试

- 文本框、表格和图表内容按页码、对象和原文稳定提取。
- 备注和图片文字不进入审核输入。
- `1、2、4` 等同组序号跳号能检出，正常序号、不同文本框序号、小数和不同层级序号不误报。
- 确定性加总只在总计与分项关系明确时执行。
- 模型候选的错误页码、错误对象、虚构原文和单边证据被丢弃。
- 同口径数据矛盾被保留，不同年份、范围、单位或累计/当期差异被过滤。
- 最终文字不出现建议字段或建议话术。
- 长结果能安全拆分且编号连续。
- PPT 模块静态或导入测试确认不依赖通用审核、文字审核或 Word 审核引擎。

### 入口与任务测试

- 审核 Bot 接收 `.pptx` 并创建 PPT 专用任务。
- 现有 `.docx`、纯文字、内参、半月报和格式审核分流保持不变。
- 同一消息重复投递不重复审核。
- worker 重启后从检查点恢复，不重复调用模型或发送结果。
- 解析失败、阶段降级和发送状态不确定时的用户提示与运维事件正确。

### 真实质量验收

实现完成后，使用经授权真实 PPT 建立第一批质量基线，至少覆盖：

- 常规文字型汇报。
- 表格和数据较多的汇报。
- 含图表的汇报。
- 长 PPT 和多处跨页口径的汇报。

人工记录有效问题、误报、漏报、页码定位准确率、耗时和模型调用次数。真实 PPT、原文和评分保存在 `M-Agent-Files/evaluations/`，不进入 Git；仓库只保留脱敏等价样本。

## 完成标准

第一阶段只有同时满足以下条件才算完成：

1. 审核 Bot 能接收单个 `.pptx` 并后台完成审核。
2. PPT 业务模块不依赖其他审核项目。
3. 每条问题都能定位到真实页码、对象和原文。
4. 跨页矛盾具备同口径双边证据。
5. 输出只有事实问题，不含修改建议。
6. 图片和备注边界向用户明确说明。
7. 自动化回归通过，现有审核流程不受影响。
8. 经授权真实 PPT 质量基线完成首轮人工验收。
