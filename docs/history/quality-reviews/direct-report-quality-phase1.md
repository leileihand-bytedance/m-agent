# 直报写作质量优化（第一轮）

更新时间：2026-07-08

## 本轮目标

先只优化 `direct_report`。

目标不是改入口，而是提升“根据任务素材写直报”的成稿质量，重点解决：

- 主线发散
- 数据抓不准
- 开头像新闻稿，不像报送材料
- 结尾空泛

## 本轮参考案例

本轮只记录从已授权样本中提炼的通用结构，不在仓库保存样本文件名、桌面路径或原文。覆盖的样本类型包括：典型案例、金融融合实践、数据要素应用、消费者教育、外贸服务、科创服务和荣誉成果。

## 从案例里提炼出的共性

### 1. 开头只有两种常见写法

- 近期事项、入选、获奖、上线类：先点事件，再点背景或意义。
- 长周期工作、政策导向类：先点政策或行业背景，再落到微众银行动作。

### 2. 正文基本都只有一条主线

- 不是把所有亮点堆在一起。
- 而是围绕“一个核心动作 + 一个支撑机制 + 一组关键结果”展开。

### 3. 数据通常只有 1 到 2 组最关键口径

常见口径：

- 金额
- 户数
- 场景数
- 机构数
- 覆盖人次
- 审批时效

### 4. 结尾很克制

- 直报通常落在“有望提供参考”“下一步持续深化”“进一步支持某方向”。
- 很少喊口号，也不会再重复前文所有成绩。

### 5. 补充材料只能做背景，不能另起业务线

- 真实案例的正文主线都来自用户核心素材本身。
- 政策或其他补充材料最多用于首段背景一句或结尾落点。
- 如果正文突然扩写到素材里没有出现的产品、业务线或成绩，直报风格就会立刻跑偏。

## 本轮实现

### 1. 新增直报写作规划器

文件：

- `skills/writing_planner.py`

作用：

- 在正式写正文前，先从材料里提炼：
  - 开头策略
  - 核心主题
  - 标题方向
  - 优先写入事实
  - 优先写入数据
  - 可借用政策背景

### 2. 在直报 workflow 中注入规划结果

文件：

- `skills/direct_report/workflow.py`

处理方式：

- `workflow` 在调用 `llm_writer` 前先生成 `planning_note`
- `planning_note` 不直接展示给用户，只作为模型写作前的内部约束

### 3. 调整模型 prompt 结构

文件：

- `app/platform/pydantic_runtime.py`
- `app/platform/builtin_tools.py`
- `skills/direct_report/prompts/draft.md`

作用：

- prompt 增加“写作规划”区块
- 明确要求模型先按规划定主线和数据，再落笔
- 禁止把规划内容原样抄进正文

### 4. 停止给直报自动补微众通用素材

本轮新增判断：

- `direct_report` 不再自动注入 `bank_materials`
- 原因是通用微众素材库里大量是“小微企业”“微业贷”“普惠金融”等长期背景，容易把不相关主题带偏
- 直报优先依赖用户素材本身，再补少量政策材料

### 5. 新增专项服务型约束，并收紧补充材料边界

本轮进一步补强：

- 新增“专项服务型”案例类型，覆盖无障碍、听障、适老化、特殊群体服务等素材。
- 写作规划中明确“正文只围绕一个核心事件展开，不另起第二条业务线”。
- 明确“政策或其他补充材料只可用于首段背景一句，不得据此引入用户素材未出现的新业务线、产品、做法或数据”。

### 6. 扩充政策主题，贴近直报案例库

当前政策意图识别已补充以下案例高频主题：

- 稳外贸和外贸金融支持
- 绿色金融与绿色转型
- 数据要素与跨境数据流动
- 科技创新和科技金融
- 金融消费者权益保护 / 特殊群体服务

目标不是让每篇直报都堆政策，而是让需要政策背景的稿件优先命中更准确的政策方向，避免被过于泛化的“小微”“普惠”等主题带偏。

### 7. 修复百家号正文解析过浅的问题

最新实测发现，部分百家号页面虽然标题能读到，但正文结构并不放在普通 `p` 标签里，而是放在文章正文区域的 `span.bjh-p` 节点中。

旧逻辑只扫普通段落时，可能只读到作者名或媒体名，例如：

- 标题读到了
- 正文只剩“新京报”等极短文本

这会直接导致：

- 写作规划抓不到“微众做了什么”
- 关键数据提取失败
- 模型被迫拿政策背景硬补正文，主线就会发虚

本轮已补上百家号正文区提取逻辑，优先读取实际文章正文，再回退到通用段落提取。

### 8. 收紧“专项服务型”直报的固定骨架

针对无障碍、听障、视障、适老化等长期开展事项，当前固定采用：

- 政策号召或现实需求
- 微众行动 / 具体举措
- 成果成效
- 下一步安排

这样做的目的，是把“长期事项”与“单次成果型事件”区分开，避免写成只有动作罗列、没有逻辑推进的稿子。

### 9. 链接未读到有效正文时，直接报错

本轮再补一个硬约束：

- 如果网页链接只读到标题、作者名、媒体名，或正文长度明显不足以支撑写作，
- `direct_report` 必须直接返回“链接暂时无法读取/未读到有效正文”，
- 不能继续补政策、不能调用写稿模型、不能依赖存量素材兜底成稿。

目标是把“素材读取失败”和“写作生成”严格隔离，避免模型在原始素材缺失时硬写。

### 10. 多主体素材统一收口到“微众银行为主”

最新实测又暴露出一个问题：

- 有些原始素材本身是多主体叙述，例如先写基金、政府部门、平台、合作方，再写微众银行。
- 如果规划层直接继承原始标题和原始首段，模型就会顺着外部新闻口径去写，导致微众银行不够突出。

本轮已新增两层硬约束：

- 写作规划明确要求“全文以微众银行为组织中心”，合作方、政府基金、平台、活动和媒体表述只作背景或协同支撑。
- 标题方向和正文主线统一改为“政策/场景背景 -> 微众银行做了什么 -> 取得什么成效 -> 下一步怎么做”，不沿用外部素材把其他主体写成全文主角。

目标不是篡改事实，而是在事实不变的前提下，把材料重新组织成符合直报件的报送视角。

### 11. 单个企业个案不能单独撑起直报

最新实测又发现一个层级问题：

- 某些素材里会带一个具体受益企业个案，例如某区一家企业获批多少授信、缓解了什么经营压力。
- 这类个案可以作为支撑例子，但如果单独起一整段，甚至被写成标题、开头或主线，就会偏离直报定位。

本轮已补两道边界：

- 写作规划和提示词明确要求：单个企业受益个案如需使用，只能压缩为主体段中的一句辅助例证，不得单独成段，不得作为标题、开头或主线。
- 如果原始素材本身只有单个企业个案，缺少机制、批量成效、政策落地或更高层级进展，`direct_report` 直接要求补材料，不再硬写成稿。

目标是让直报始终落在“机制、模式、批量成效、政策落地”这个层级，而不是掉到单个企业故事层面。

### 12. 初稿语义审查常态化

此前 critic 只在标题、字数、小标题、个案单独成段等确定性硬规则失败后触发。这样会漏掉一类问题：稿件表面格式合规，但仍然像新闻通稿、主线发散、政策背景挂得生硬或微众银行主角不够突出。

本轮已调整为：

- 每次直报初稿生成后，都调用 `critic` 做语义质量审查。
- critic 重点检查直报文体、单主线、主体视角、政策挂接、事实边界和宣传腔。
- 整体仍像新闻稿、通稿、活动报道或材料摘编时，按 hard violation 处理；个别词句轻微宣传腔可按 soft 处理。
- 由于真实测试中发现 critic 会增加模型调用次数和结构化输出失败概率，当前默认模式改为 `advisory`：照常审查并把风险写入返回提示，但不因为 critic 的 hard violation 自动重写。
- 如需恢复强干预，可在环境变量中设置 `M_AGENT_DIRECT_REPORT_CRITIC_MODE=rewrite`，critic 返回 hard violation 时才会把问题和修改建议整理成 `revision_feedback`，自动重写一次。
- 如需排查模型或链路稳定性，可设置 `M_AGENT_DIRECT_REPORT_CRITIC_MODE=off`，临时关闭 critic。
- 重写后仍保留确定性规则检查，避免为了修语义问题引入标题、字数、小标题等硬规则问题。
- critic 是质量增强层，不是主写作链路的硬依赖。如果 critic 模型调用或结构化输出偶发失败，应降级为“未发现语义违规”，让已经生成的初稿继续返回，避免质量审查层导致整次直报失败。

目前三种模式的定位：

- `advisory`：默认。适合真实企业微信测试，稳定性优先。
- `rewrite`：适合离线质量压测，质量干预更强，但耗时和失败概率更高。
- `off`：适合排查网页读取、路由、模型主写作等基础链路问题。

目标是把“写完后只查格式”升级为“格式 + 语义质量”双层把关，同时不让质检层拖垮主写作链路。

### 13. 为综合类素材新增“综合进展型”写法

最新真实测试又暴露出一个新问题：

- 大部分直报确实适合“一事一议”。
- 但少数素材本身是年度盘点、阶段性综述、综合进展类文章，里面会同时出现多个板块。
- 如果仍然硬套“只保留一个细节主线”，模型就容易抓住某个近期产品或单一动作，把整篇稿子写偏。

本轮新增约束：

- 写作规划器新增“综合进展型”案例类型。
- 只有当素材明显属于综合进展类时，才允许采用“一个总主题 + 2-3 个并列板块”的写法。
- 标题方向和核心事件不再允许被某个单一产品、活动或近期细节劫持。
- prompt 和 critic 同步放宽到这个特例场景，但普通直报仍保持单主线要求。

目标是把“例外场景”显式规则化：既不把综合稿硬压成单一细节，也不把普通直报放开成多点罗列。

### 14. 直报主体称谓

用户确认直报中的主体称谓规则：

```text
全部直接使用“微众银行”
```

不得写：

```text
深圳前海微众银行
深圳市微众银行股份有限公司
微众银行股份有限公司
全称（以下简称“微众银行”）
我行
```

当前已把该规则加入两层约束：

- `skills/direct_report/prompts/draft.md`：写作提示词和输出前自检明确要求直报全篇直接使用“微众银行”。
- `skills/direct_report/guardrails.py`：确定性校验会拦截全称、错误全称和“以下简称”等写法，并作为 hard violation 触发重写反馈。

### 15. 默认强化句间衔接与结尾抬升

最新讨论又明确了一点：

- 直报的问题不只是“有没有主线”，还包括句与句之间是否顺、段与段之间是否递进。
- 如果政策背景写完后直接跳数据、跳产品细节，读起来就会断。
- 结尾如果只把眼前这件事收住，也会显得就事论事、站位不够。

本轮新增默认要求：

- 写作规划器新增“衔接要求”，对政策背景型稿件明确要求使用“在此背景下，微众银行……”或“微众银行积极响应相关部署……”这类承接句，体现“中央有部署，微众有响应”。
- 写作规划器新增“段落推进”，统一要求各段按“背景/部署 -> 微众响应/动作 -> 成效 -> 下一步”递进，段首承担承上启下作用。
- 写作规划器新增“结尾抬升”，统一要求先写微众银行下一步安排，再自然抬升到与主题一致的更高层方向，例如提升小微企业融资可得性、稳订单拓市场、做好金融“五篇大文章”等。
- draft prompt 和 critic 也同步增加这套要求，避免模型只做到“句子能通”，做不到“逻辑顺、层次清”。

目标是把“文字细节能力”从个别提示升级为直报默认写法，而不是每次都靠人工在改稿阶段提醒。

### 16. 标题两段式连接符统一改为逗号或冒号

用户又补充了一个细节要求：

- 无论是直报还是简报，只要标题不是一个完整短句，而是明显分成两层意思，
- 中间都不要再用空格硬断开，
- 而要根据语义改用中文逗号或冒号连接。

本轮对直报已补两层约束：

- `skills/direct_report/SKILL.md` 和 `skills/direct_report/prompts/draft.md` 明确写入标题连接符规则。
- `skills/direct_report/guardrails.py` 新增确定性校验：如果标题是两段式结构却用空格硬断开，直接判为 hard violation，提示改成中文逗号或冒号。

目标是统一标题观感，避免出现“前半句 + 空格 + 后半句”的生硬分隔。

简报与直报不同：简报正文第一次提到主体时使用“深圳前海微众银行（以下简称“我行”）”，后文统一使用“我行”。

### 14. 政策挂接闸门和开头策略

用户评审 4 个直报回归样本后确认：政策背景不是越多越好。直报写作优先保证准确、自然、有必要，不得为了显得正式而强行挂接政策。

当前规则：

- 案件、活动、获奖、直播、判决等时间节点稿件，默认采用“直入主题型”开头。
- 上述时间节点稿件默认不额外注入政策库材料，也不写具体政策文件名称。
- 产品、机制、长期服务、政策落地类素材，才允许在写作前补充政策背景。
- 具体政策文件名称只能来自用户素材或补充材料；没有明确来源时，不写具体政策名。
- 如果只需要政策语境，可以写宽泛导向，例如“围绕稳外贸、促融资、优化小微企业金融服务等部署”，不要编造文件名。

当前已把该规则加入三层约束：

- `skills/writing_planner.py`：通过 `should_add_direct_report_policy_materials(...)` 判断是否允许补充政策库材料。
- `skills/direct_report/workflow.py`：只有闸门允许时，才调用政策库相关工具。
- `skills/direct_report/prompts/draft.md` 和 `skills/direct_report/prompts/critic.md`：写作和审查都明确禁止编造或强行挂接具体政策名称。

后续质量优化重点不再是简单增加政策库数量，而是提高“是否该用政策、用哪类政策、写到什么程度”的判断能力。
