# M-Agent 底座区规划

## 定位

底座区负责 M-Agent 的公共运行能力。它不直接决定一篇直报怎么写、一个文档怎么审，而是负责：

- 接收企业微信消息
- 判断用户要做什么
- 检查用户是否有权限
- 找到对应 skill
- 给 skill 分配允许使用的工具
- 保存任务记录
- 把结果安全返回给用户

一句话概括：

> 底座区是 M-Agent 的运行框架，功能区是 M-Agent 的业务能力。

## 为什么需要单独建设底座区

现有 M-Agent 已经有审核、写作和早期 agent 代码，但它们更多是独立模块：

```text
app/review/     # 审核 Bot
app/writing/    # 写作 Bot
app/rewrite_bot/ # 独立材料润色 Bot 入口
```

如果继续每新增一个能力就做一个独立 Bot，后续会出现几个问题：

1. 用户入口分散，不知道该找哪个 Bot。
2. 每个功能都要重复处理企业微信、文件、权限、日志。
3. 多轮改稿和任务上下文难以复用。
4. 安全边界不统一，容易让某个功能越权访问本机资源。
5. Codex / Claude Code 后续维护时，缺少清晰边界。

因此需要新建底座区，把公共能力沉淀出来。

## 推荐技术路线

底座区推荐采用：

```text
uv 管理的 Python 3.13.14 和项目独立 .venv
+ Pydantic AI
+ MCP / 现成工具
+ 文件化 Skill Registry
+ 后续按需引入 LangGraph
```

项目依赖由 `pyproject.toml` 声明、`uv.lock` 锁定，所有底座入口统一通过 `uv run --locked ...` 启动。这样写作、审核、运维和管理后台不会因为终端默认 Python 或其他项目升级依赖而发生变化。

### Pydantic AI 的角色

Pydantic AI 负责：

- 调用大模型
- 注册工具
- 约束结构化输入输出
- 把每个 skill 的执行过程封装成可测试的 Python 逻辑

当前已接入：

```text
app/platform/pydantic_runtime.py
skills/direct_report/schema.py
```

`direct_report` 已经通过 Pydantic AI Agent 返回结构化 `DirectReportResult`。

选择它的原因：

- 和当前 M-Agent 的 Python 代码一致，迁移成本低。
- 适合被 Codex / Claude Code 通过自然语言继续开发。
- 比完整平台更轻，不需要大量界面配置。
- 比个人助手框架更容易控制工具权限。

### MCP / 现成工具的角色

MCP 和现成库用于接入基础能力：

- 联网搜索
- 网页读取
- Word 读取
- PDF 读取
- PPTX 读取
- 知识库或数据库
- 其他外部服务

但 MCP 和工具必须经过底座区授权，不能直接暴露给用户。

公共联网搜索当前按供应商分流：DeepSeek 通过 Anthropic Messages 的服务器端 `web_search_20250305` 执行，MiniMax 旧搜索端点仅作显式兼容。模型生成地址不能直接当作通用搜索地址；未知供应商由底座拒绝，Skill 只能获得统一的标题、链接、摘要和来源类型结果。

### LangGraph 的角色

第一阶段不建议直接引入 LangGraph 作为主框架。

后续当出现复杂长任务时，再引入 LangGraph，例如：

- 多轮改稿
- 长流程审核
- 多步骤写作
- 人工确认后继续执行
- 任务中断后恢复

## 底座区模块说明

### 1. 企业微信入口

当前位置：

```text
app/platform/gateway/wecom.py
```

职责：

- 接收企业微信文本、链接、文件消息。
- 下载用户上传的文件。
- 把用户、群、消息、附件整理成统一请求。
- 把最终结果返回企业微信。

原则：

- 继续优先复用当前已经跑通的企业微信 SDK 经验。
- 不在入口层写具体业务逻辑。
- 入口层只做接收、回复、转发和基础校验。

当前已完成文本消息核心：

```text
extract_text_message     # 从企业微信消息结构中取出文本和用户 ID
handle_text_frame        # 调用平台 runner
format_text_reply        # 把 PlatformResult 整理成回复文本
handle_text_frame_with_app # 直接调用 PlatformApp
```

当前写作 Bot 还新增了一个轻量素材入口页：

```text
企业微信欢迎语
  -> 引导用户直接发送链接，并说明是写简报还是写直报
  -> 多个素材文档建议先整合成一个文档后再发送

本机素材页
  -> 收集文件、链接和补充要求
  -> PlatformApp 结构化提交
  -> 结果主动回企业微信对话
```

当前写作 Bot 已经通过 `app/writing/bot.py` 接入真实 AiBotSDK 长连接，并调用 `PlatformApp`。材料润色另设 `app/rewrite_bot/` 作为过渡期独立入口，但业务逻辑仍只保留在 `skills/rewrite/`，底座、权限和工具层全部复用。该入口自己的纯文字 intake 只负责保存“尚未说明要求的原文”并追问一次，不包含润色规则，也不能接收其他 Skill。长期方向仍是统一入口，不把每个新 Skill 都复制成一套独立业务实现。

写作 Bot 当前还具备短任务组装 v1：

```text
先发链接/文字/文件 -> 后说“写简报/写直报”
先说“帮我写简报” -> 连续发多个材料 -> 回复“开始写”
直接粘贴文字 -> 可选择独立 `rewrite` 润色
先贴新原文 -> 再单独补一句“帮我整体润色一下” -> 也会新开 `rewrite`
说明“做综合调研整合” -> 连续发送提纲和部门文件 -> 回复“开始写”
综合调研要求确认提纲 -> 保留原文件和会话 -> 用户回答后自动续跑
```

写作状态机由 `app/writing/intake.py` 实现，审核状态机由 `app/review/intake.py` 实现；两者共同复用 `app/platform/intake.py`。公共层统一匿名用户目录、原子状态、文件暂存、重启恢复、TTL 清理、防目录外文件引用、文件数量和总大小校验，并统一材料引用、`wait/submit/cancel/bypass` 动作和结构化提交。具体“写什么”“是否格式审核”“哪份是正文”仍留在各模块。

写作入口当前每次任务最多接收 10 份文件，文件总大小仍不超过 20MB。该调整只放宽文件数量，不改变格式白名单、单文件大小、任务隔离和统一文档安全解析规则；审核 Bot 的 2 至 5 份联合审核范围保持不变。

### 2. 身份和权限

当前位置：

```text
app/platform/identity.py
config/platform-policy.example.yaml
```

职责：

- 识别企业微信用户 ID。
- 判断用户是否在白名单。
- 判断用户所在群是否允许使用某些能力。
- 控制不同用户可调用的 skill 范围。

第一阶段可以简单做成配置文件：

```yaml
allow_unknown_users: false
default_allowed_skills: []

users:
  user_a:
    allowed_skills:
      - direct_report
      - writer1
      - writer2
      - research_synthesis
  user_b:
    allowed_skills:
      - review
```

### 3. 意图识别和路由

职责：

- 判断用户是要写直报、写简报、审核、改稿，还是提出了范围外问题。
- 如果不确定，向用户追问。
- 如果不属于已有能力，拒绝执行。

重要原则：

> 意图识别只能在已登记 skill 中选择，不能让模型自由发明新能力。

### 4. Skill 注册表

职责：

- 扫描 `skills/` 目录。
- 读取每个 skill 的说明、触发条件、输入输出、允许工具。
- 给路由层提供可用能力清单。
- 给运行层提供 skill 执行入口。

注册表应该只接受仓库中已提交、已测试的 skill。

### 5. 运行层

职责：

- 加载 skill。
- 组装 prompt 和上下文。
- 调用 Pydantic AI agent。
- 调用允许的工具。
- 校验输出结构。
- 返回结果。

运行层不直接决定业务风格，业务风格放在 skill 中。

### 6. 受限工具层

基础工具包括：

- `web_reader`：已实现，读取用户给出的网页链接。
- `word_reader`：已实现，读取用户本次任务目录内的 Word 文件。
- `pdf_reader`：已实现，读取用户本次任务目录内的 PDF 文件。
- `document_reader`：已实现，统一安全读取 DOCX、PDF 和 PPTX，返回适合模型使用的标准材料片段，同时把完整解析结果写入任务 `work/documents/`。主文档宏格式、真实 VBA 项目、其他 `macroEnabled` 部件、VBA 关系以及非规范或重复的主部件声明继续拒绝；只对 PPTX 中位于 `ppt/embeddings/`、并精确声明为标准 XLSB 类型的 `.xlsb` 图表数据做兼容。PPT 独立审核不会单独解析这类嵌入工作簿，也不会把其内容送入模型。PPTX 文本保留段落层级缩进，图表底层数值读取失败会留下带幻灯片位置的结构化告警。扫描 PDF 只对明确标记的页面按需 OCR；PDF/PPTX 页面渲染按工作流显式启用。DOCX 内嵌图片不默认 OCR，在原位置生成可传给模型的人工评估提醒；长材料抽样会强制保留提醒。
- `llm_writer`：已实现，通过 Pydantic AI 调用模型生成文稿。
- `search`：已实现，调用搜索 API 返回标题、摘要、链接和来源类型。
- `policy_research` / `policy_materials` / `policy_search`：已实现，提供共享政策挂靠判断、政策知识库材料包和底层检索。
- `review_engine`：待实现，后续包装现有审核能力。

工具必须遵守：

1. 只能访问本次任务材料。
2. 不能读取 Mac 上的任意目录。
3. 不能执行系统命令。
4. 不能把密钥、配置、日志、历史文件暴露给用户。
5. 每次工具调用都要记录日志。

### 7. 文件隔离

底座运维通知采用“当天实时、历史汇总、同类去重”规则：运维 Bot 只处理当天新事件，前一工作日事件由工作日报汇总，不在重启后逐条补发；同一天内来源、级别、主题和详情完全相同的事件只通知一次，重复次数仍保留在日报中。这样既保留故障记录，也避免重连抖动、迁移或周一重启造成告警刷屏。

每个任务应有独立任务目录：

```text
../M-Agent-Files/tasks/writing/
└── 2026/07/20260703-0001/
    ├── input/
    ├── work/
    ├── output/
    ├── meta.json
    └── status.json
```

skill 和工具只能访问当前任务目录。

公共文件与文档处理继续统一使用桌面 `M-Agent-Files/`：尚未开始处理的文件暂存在 `runtime/intake/`；任务原件进入 `input/`；DOCX/PDF/PPTX 的完整解析结果进入 `work/documents/`，提取图片、页面渲染和 OCR 中间产物进入对应任务 `work/`；最终报告、标注或生成文件进入 `output/`。这些运行文件都不进入 Git。

当前写作材料格式范围仍是 Word、PDF 和 PPTX，写作统一文档服务不接收 HTML；现有网页链接读取能力保持不变。独立审核 Bot 另行支持单个 `.html/.htm` 的静态可见文字与页面内部数据一致性审核：不执行脚本、不联网加载资源、不进入公共文档服务或 Word 多文件组装。

统一文档服务当前边界：原生文字 PDF 可按页解析，疑似扫描页可按需 OCR；PDF/PPTX 可显式逐页渲染，默认最多渲染 200 页、OCR 50 页，并限制单图、总图容量和总处理时间。PPTX 视觉版式判断、问题规则和批注回传仍是审核业务能力，不因底座能渲染页面而自动具备。

当前已由 `app/platform/storage.py` 实现，实际目录名使用时间戳和随机短 ID，避免并发冲突。

`status.json` 是不含材料正文的任务状态索引。同步写作任务创建时为 `processing`；进入持久队列的直报/简报先记为 `queued`，worker 执行时转为 `running/processing`，随后按实际结果更新为 `completed`、`needs_input` 或 `failed`。处理状态与企业微信交付状态分别记录，队列主动发送成功后才记为 `delivery_status: delivered`；发送状态无法确认时停止自动重发并告警。本机控制台只读取该索引和必要的文件存在性，不读取结果正文来做总览统计。历史数据使用 `uv run --locked python scripts/backfill_task_status.py` 预演，确认后加 `--apply` 补齐。

- 受限工具网关。
- 网页读取，以及 DOCX/PDF/PPTX 统一文档读取工具。
- 企业微信文本消息核心适配。
- 平台应用服务 `PlatformApp`。
- 用户/skill 权限控制。
- 每次请求独立任务目录和结果记录。
- 同一用户、同一入口的上一稿改稿 v1。
- CLI 配置检查和本地消息入口。
- 写作 Bot 真实企业微信文本入口。
- 独立材料润色 Bot 入口：注册表白名单固定为 `rewrite`，只接收直接粘贴文字，文件和链接明确拒绝；任务、待确认原文和会话目录与原写作 Bot 隔离。只有原文时先追问修改方向，收到要求后再执行；同条消息含要求或上一版续改时直接处理。待确认原文默认 30 分钟有效并支持 Bot 重启恢复。微众相关原文可通过 Skill 自身的关联闸门读取本地 `bank_materials` 做名称和标准表述核对，普通材料不检索，且不得借语料新增事实。
- 写作 Bot 短任务组装 v1：支持链接、文字、Word/PDF/PPTX 跨消息组成一次直报/简报请求，也支持把 1 份现成提纲和多部门文件组成一次综合调研整合请求；待组装文件和待澄清任务可在 Bot 重启后恢复，用户回答后沿原材料续跑。`rewrite` v1 只接收直接粘贴文字，并支持“原文在前、润色要求在后”的新任务写法，不会误接到上一条直报/简报改稿。
- 综合调研 Word 结果交付 v1：先生成结构化材料台账，再按提纲问题跨部门归并正文；成稿写入当前任务 `output/`，按现有公文基本格式生成，不嵌入源图片，并由写作 Bot 作为企业微信文件返回。生成后独立校正标题编号、来源标签、提纲一级主题和重复图片提醒；附件路径和大小在发送前再次校验。
- 审核 Bot 短任务组装 v1：Word 格式审核支持文件和指令两种顺序；直接发送 1 份 `.docx` 自动走单文件审核，连续发送 2 至 5 份自动走联合审核，无需前后指令；保留取消、主文件确认和持久化恢复，不按上传顺序默认正文。单份 `.pptx` 直接进入独立低级错误审核任务，不参加 Word 组装、格式或联合审核。
- 本机管理后台。
- 搜索工具。
- 公共任务暂存内核：写作和审核共用安全文件持久化、原子状态、TTL、恢复和限制校验。
- 公共任务组装协议：统一材料引用、`wait/submit/cancel/bypass` 动作和结构化提交，业务分流仍由写作、审核适配层负责。
- 公共附件交付：写作和审核统一使用任务目录校验、串行上传、动态超时、完整重试、状态记录和运维告警；超限或失败时返回“处理编号”供人工取回。后台任务编号始终保留，正常受理和重复提交提示不向用户展示。
- 持久化后台任务内核：SQLite 队列、消息幂等、分级并发、租约与 fencing token、心跳、周期恢复、取消、待补充恢复和状态版本已完成。
- 持久任务业务接入：纯文字、单个通用 Word、内参、半月报、公文格式、单个静态 HTML 和单份 PPTX 使用审核专用队列；直报、`writer1`、`writer2` 使用独立写作队列。审核输入在入队前冻结到任务目录，写作任务入队前创建正式 job；两边都按处理和主动发送检查点恢复。PPTX 结果可在同一检查点保存为多段文字，按顺序全部发送后才标记交付完成。
- 文档增强：扫描 PDF 按需 OCR，PDF/PPTX 显式页面渲染和资源限制已完成。

仍未完成：

- `research_synthesis` 尚未切换；它包含 Word 附件交付，需单独设计和验证恢复检查点。多文件联合审核也继续使用原路径，因为当前单结果检查点不能安全表达多个输入和多个返回文件。
- 覆盖直报、简报、审核等能力的统一企业微信入口。
- 审核能力包装为 `review` skill。
- 复杂多轮任务上下文和人工确认。

审核旧模块已接入统一数据根目录，任务写入 `../M-Agent-Files/tasks/review/YYYY/MM/`；原件和系统生成文件不再混放。统计同时识别历史 `meta.md`、当前 `meta.json` 和 `output/report.md`，避免历史审核任务因元信息格式不同而漏计。

### 8. 会话和任务记录

第一阶段已经支持短任务：

```text
用户发链接 -> 生成结果 -> 返回
```

当前已支持“上一稿改稿 v1”：

```text
用户发链接 -> 生成直报/简报初稿
用户继续说“再压缩一点” -> 系统读取同一用户、同一入口最近一次输出 -> 回到上一轮 skill 修改上一稿
```

这个能力只使用上一轮 M-Agent 生成的标题、正文和来源，不读取历史任务目录里的原始文件，也不会把本机其他文件暴露给用户。

后续支持复杂多轮任务时，需要进一步记录：

- 用户是谁
- 用户名和企业微信 userid 的对应关系
- 上一次调用了哪个 skill
- 输入材料是什么
- 输出结果是什么
- 用户这次修改要求是什么

可以先用文件或 SQLite，不急于引入复杂数据库。

当前用户名映射已经作为底座公共能力落在：

```text
app/platform/user_registry.py
../M-Agent-Files/runtime/users/review_users.yaml
```

写作 Bot 和审核 Bot 共用同一份映射表。写作链路会在任务记录、会话记录、开发期对话日志和运行输出中同时保留 `sender_name` 与 `sender_userid`。该文件位于 Git 仓库之外，属于本机敏感运行数据。

### 9. 安全边界

底座区必须坚持：

1. 企业微信用户不是可信用户。
2. 用户发来的网页、文档、文字都可能包含诱导模型越权的内容。
3. 安全不能只靠 prompt。
4. 所有工具必须通过底座授权。
5. 对外只暴露已登记 skill，不暴露本机助手能力。

## 与现有功能的关系

```text
app/review/   # 继续作为现有审核 Bot 运行

```

新底座最小闭环稳定后，按可控范围逐个接入：

1. 审核 Bot 内七类单项审核已经迁移，不改变独立入口；新增的 PPTX 审核是独立业务包，不导入其他审核引擎。
2. 直报和两类简报写作已经迁移。
3. 综合调研和多文件联合审核分别评估多文件、多结果检查点后再切换。
4. 新增任务类型继续逐类证明可恢复性，不整体切换。
5. 最后考虑长期记忆能力。如需参考早期领导风格沉淀实验，只查看 `archive/inactive-2026-07-04/`，不要恢复旧入口。

## 底座区完成标准

底座区只有满足以下条件才算可用：

1. 能统一接收企业微信消息。
2. 能按用户权限过滤 skill。
3. 能识别意图并路由到已登记 skill。
4. skill 只能调用声明过的工具。
5. 工具只能访问当前任务材料。
6. 每个任务有日志和输出记录。
7. 测试能覆盖路由、权限、工具限制和至少一个真实 skill。
