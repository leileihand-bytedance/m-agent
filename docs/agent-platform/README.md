# M-Agent 公共底座

本文档说明 `app/platform/` 对所有入口和 Skills 提供的公共能力、接口边界和扩展约束。完整实现分层见 [整体架构](../development/architecture.md)。

## 职责

底座负责：

- 把入口消息转换为统一请求。
- 识别意图并从已启用 Skill 中路由。
- 校验用户和 Skill 权限。
- 管理多消息材料组装、会话和任务状态。
- 为支持续改的 Skill 提供统一改稿计划、范围保护和结果自检。
- 安全解析 DOCX、PDF 和 PPTX。
- 通过 `ToolGateway` 限制 Skill 可用工具。
- 调用 Pydantic AI 和 Skill workflow。
- 提供持久任务、幂等、并发、租约、恢复和取消。
- 安全交付文字和附件，并记录不含正文的状态和运维事件。

底座不负责：

- 直报、简报、综合调研、深银协动态、内参周报、润色或审核的业务规则。
- 为某个 Skill 私自扩大文件或工具权限。
- 读取任务目录以外的任意本机文件。
- 向外部用户暴露 shell、密钥、日志或历史任务。

## 与相邻模块的逻辑边界

`app/platform/` 是当前公共代码包，不代表其中每个模块在产品架构上都属于“智能体底座”。项目总览按职责作如下区分：

- `智能体底座`：入口适配与权限、意图路由与 Skill 注册、Agent 运行和工具授权、会话版本、任务组装、持久执行、任务状态与文件边界。
- `共享工具服务`：统一文档服务、网页读取与联网搜索、结果与附件交付。这些模块由底座授权给业务能力调用，但不是路由或编排本身。
- `知识资产`：政策知识库和微众银行信息库。知识资产为业务提供事实，不执行任务；其可视化筛选和维护属于知识库治理。
- `领域公共组件`：审核共享核心、写作末端成稿审核等业务域内部复用组件。它们包含领域判断，不进入通用底座。
- `管理与治理`：管理台、运维告警、日志指标、数据保留和研发交付闸门。它们观察或约束业务运行，不是企业微信用户的业务入口。

后续移动物理目录不是落实上述边界的前置条件。新增代码时先判断逻辑职责，再沿用现有包结构；只有目录混用实际造成依赖倒置或权限绕过时，才单独安排迁移。

## 公共协议

### Skill 注册

每个 Skill 通过 `skills/<skill_id>/config.yaml` 声明：

- ID、名称和触发词。
- workflow 入口。
- 允许使用的工具。
- 输入输出类型。
- 是否支持继续改稿。

底座只加载已登记且启用的 Skill。

### 工具授权

Skill 只能调用 `config.yaml` 中声明且由底座注册的工具。工具必须验证任务目录和输入类型，不能接受任意本机路径。

模型调用同样经过工具授权。`llm_writer` 用于生成业务结果，`llm_planner` 用于返回 Skill 自己声明的结构化规划；两者复用同一受控 Pydantic AI 运行器和可靠性策略，规划别名不具备额外文件、网络或模型权限。

### 材料组装

公共 intake 使用 `wait`、`submit`、`cancel` 和 `bypass` 动作描述入口决定。底座负责安全暂存、恢复、过期和文件限制；具体“何时开始写”“哪份是正文”等业务判断仍由入口或 Skill 负责。

材料组装完成后，任务关系层先判断材料属于哪项任务，再交给 Skill：`supplement` 表示补充，`replace` 表示替换旧事实，`reference` 表示只作参考，`new_task` 表示独立新任务。上传文件或链接本身不再直接等同于新任务。

### 任务关系和改稿

`task_relations.py` 按 `channel + userid` 隔离任务卡片。每张卡片保存标题、Skill、状态、版本、结构摘要、材料台账、待回答问题和父任务；同一用户可以同时保留多项任务，不再依赖一个全局“最近稿件”。

公共关系固定区分续改、补充材料、派生新稿、独立新建、回答追问、切换、取消和需要确认。任务序号、明确标题或文种优先于当前稿；“再改、继续改、接着改、这个直报/简报”、对当前稿“输出/导出 Word、生成正式文档”，以及明确修改日期、期号、责任编辑等文档字段的表达默认指向用户当前稿。只有剩余模糊表达才调用 Pydantic AI 结构化分类器。模型只看到当前用户最多 8 项任务的标题、状态、是否为当前稿和受限摘要，目标不存在或置信度低于阈值时必须追问，不能自行创建任务编号或扩大工具权限。关系分类使用独立的一次调用和最多 15 秒超时，不沿用长篇写作的重试预算。

追问、原始要求和材料引用保存在 `runtime/task-relations/task-relations.sqlite3`。用户回答稿件名称或纠正为“另写一份”后，系统恢复原要求和已上传材料，不要求重传。目标稿仍在排队或运行时先等待当前版本完成；不同稿件可以继续排队，同一用户执行层仍保持单任务并发，避免同一人的多稿同时抢占模型和材料。

任务关系层先回答“改哪一稿”，`RevisionEngine` 再回答“这一稿改哪里、哪些内容不能动”。公共引擎把要求解析为 `RevisionPlan`，标题或指定段落的未点名内容由代码恢复；篇幅、数字保持和指定删除项会在生成后检查，失败时只进行一次带反馈修正。直报、简报和材料润色共用该协议，文种语气、结构、事实边界和质量规则仍归对应 Skill。

会话版本除标题、正文和来源外，还可保存受限的结构化 `document_metadata`，由具体 Skill 定义允许字段。持久任务结果还保存经过白名单清洗的 `revision_plan` 供开发期追踪，但普通用户回复不展示该内部字段。平台只负责在当前用户、当前任务的版本链中透传和恢复，不解释业务含义；缺少这些字段的历史版本按空值兼容。

### 文档服务

统一文档服务接收任务目录内的 DOCX、PDF 和 PPTX，输出标准文档结构。扫描 PDF OCR、PDF/PPTX 页面渲染和图片提取均受页数、时间、像素和容量限制。支持读取不等于支持视觉审核或版式修改。

公共 Word 目录终稿服务只接受调用方指定任务输出目录内的 `.docx`。它把文件复制到 Microsoft Word 自有沙盒中的固定暂存目录，通过固定 AppleScript 更新一个自动目录，随后校验预期标题、页码、`PAGEREF` 和 `_Toc` 书签，再原子写回任务文件；不把任务路径拼进脚本，也不接受外部用户提供脚本或命令。Word 不可用、超时或目录缓存不完整时返回失败，不能交付空目录文件。暂存副本在成功或失败后清理，Word 保存产生的作者元数据在写回前统一净化。

### 持久任务

任务执行器提供消息幂等、排队、并发限制、租约与 fencing token、心跳、取消和进程恢复。每种业务任务仍需单独登记 handler，并验证模型调用、文件生成和外部发送的幂等性。

当前已登记的持久任务包括审核模块八类能力，以及直报、简报、综合调研、深银协动态和内参周报。简报只登记 `writer1` 一个内部 handler ID；历史持久化记录中的退役 ID 在读取时归一到 `writer1`，不再形成新的任务类型。简报正式 Word、综合调研、联合审核、深银协动态和内参周报的文字与附件分别记录交付检查点；综合调研的 Word 是独立附件，内参周报初次生成只交付精简 Markdown 核对稿，JSON 溯源记录留在任务内部，用户明确批准同一版本后的洁净版 Word 是另一项独立附件。已确认成功的交付项不会在进程恢复后重复发送。管理台直接根据真实注册表展示持久队列状态；即时润色仍是实时多轮交互。

文本和附件共用同一套精确交付状态：`confirmed_delivered` 表示收到成功回执，`confirmed_not_delivered` 表示收到非零回执或本地明确拒绝，`delivery_unknown` 表示发送已经发起但未取得可判断回执。SDK 1.0.7 没有可用的发送状态查询接口，因此未知状态不会自动重发。本机管理员可在管理台查看待人工处理项，并在人工核实后按原检查点重发、确认已送达或关闭；命令行 `delivery_recovery` 保留为排障入口。两种入口都不向企业微信用户和 Skill 开放，也不会重新运行模型。

### 模型调用可靠性

`model_reliability.py` 把写作、审核和模型搜索的供应商异常归一为超时、限流、鉴权、连接、服务不可用、无效结构化输出和一般调用失败。写作 Pydantic AI 与公共搜索使用可配置的超时和最多三次有限调用；审核各业务阶段继续遵守原有固定调用预算，但异常使用同一套安全分类。原始供应商错误不作为用户提示或任务安全错误码。系统不会自动切换到其他模型供应商，避免结果口径在用户不知情时变化。

### 数据和日志

任务、会话、队列、日志、知识库和运维状态位于 `M-Agent-Files/`。Git 仓库只保存代码、脱敏测试样本和静态配置示例。

任务关系库另存不含用户消息正文的判断指标，包括追问、追问恢复、用户纠正和语义判定次数，用于前期校准自动执行边界。正文仍只保存在受限任务目录和按权限启用的开发期对话日志中。

`app/platform/runtime_environment.py` 统一区分生产和测试：生产 Bot 只能从 `main` 启动；任务分支只能使用专用测试 Bot 和独立测试数据根目录。测试模式不回退生产凭据，并在连接企业微信前确认所有运行数据路径都位于测试根目录。这是代码硬边界，不由 Skill 或模型自行判断。

## 安全边界

1. 用户消息、网页和文档全部视为不可信输入。
2. 安全依赖代码校验和目录隔离，不能只靠 prompt。
3. 网页读取拒绝本机、内网、云元数据和非 HTTP/HTTPS 地址；发布日期同时识别常见新闻元数据和研究报告常用的 `citation_publication_date`、`DC.date`。权威站点的非标准模板若在公共页头后提前闭合 HTML，工具会在原始页面确有段落、但主解析器没有取得段落时切换解析方式，恢复后续文章正文和可见发布日期。公开 HTML 列表返回受限数量的带日期链接；公开 JSON 索引兼容常见嵌套记录，只返回受限的链接元数据和标量记录；JSON 原文只归一标题、日期和正文。工具不自动打开列表链接，后续读取仍由 Skill 的来源策略控制。
4. 文档读取拒绝目录越界、宏和异常压缩包等高风险输入。
5. 外部模型只接收当前任务需要的最少材料。
6. 日志、状态索引和控制台默认不读取或展示材料正文。

## 扩展要求

- 新公共能力放入 `app/platform/`，并提供平台测试和架构说明。
- 新业务能力放入 `skills/`，不能把业务判断塞进入口或底座。
- 复杂流程只有在现有协议不足且有真实场景证明时，才评估 LangGraph 等状态图框架。
- 底座当前行为改变时更新本文档或整体架构；开发历史写入月度日志，不追加到本文档。
