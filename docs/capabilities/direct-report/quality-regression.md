# 直报质量回归方法

本文档规定直报成稿质量如何重复评估。业务写作规则仍以 `skills/direct_report/SKILL.md` 为准；本文不保存某次生成稿、任务编号或历史测试流水。

## 固定样本

脱敏配置位于：

```text
tests/fixtures/direct_report_quality_cases.json
```

当前样本使用用户确认可用于测试的公开网页，覆盖政策挂接风险、科技金融、风险治理、稳外贸以及网页噪音等场景。公开链接可以保留在 fixture，真实生成结果只保存在 `M-Agent-Files/`。

## 自动化部分

```bash
uv run --locked pytest tests/test_direct_report_quality_regression.py -q
```

自动化测试负责检查：

- 固定样本仍可被识别为直报任务。
- 人工评测字段和样本元信息完整。
- 主体称谓、标题、字数、小标题、个案成段等可确定判断的规则不回归。
- 已确认的错误模式有脱敏等价样本保护。

自动化测试不能独立判断文章是否像直报、政策挂接是否自然、事实取舍是否合适或初稿是否真正可用。

## 人工评测

每次影响直报提示词、规划器、critic、政策研究、网页读取或改稿行为时，对固定样本做前后对照。每篇按以下维度记录 `0` 到 `3` 分和一句主要问题：

| 维度 | 判断内容 |
| --- | --- |
| 可用性 | 能否作为初稿继续修改 |
| 直报风格 | 是否仍像新闻稿、宣传稿或摘要 |
| 主体稳定 | 是否始终以微众银行为主体 |
| 政策挂接 | 是否准确、必要、自然且可追溯 |
| 事实可靠 | 是否忠于素材，没有编造或混入口径 |
| 修改成本 | 用户需要大改、中改还是小修 |

评分含义：

```text
0 = 不合格
1 = 勉强可用，需要大改
2 = 基本可用，需要中等修改
3 = 可用，只需小修
```

## 执行规则

1. 先运行自动化测试，再在获授权环境生成当前版本结果。
2. 前后版本使用同一批样本和同一评测维度；模型波动与代码差异分开说明。
3. 真实输出、任务编号和逐篇评语保存在 `M-Agent-Files/evaluations/` 或任务目录，不写入当前文档。
4. 只把可泛化的结论转成 Skill 规则、确定性校验或脱敏自动化样本。
5. 某次评测的功能变化、关键验证和剩余边界写入月度开发日志。

## 新增样本

优先增加：

- 用户明确认为不可用或方向错误的失败样本。
- 用户确认方向较好的对照样本。
- 链接读取失败或正文抽取噪音明显的样本。
- 政策挂接容易生硬或事实口径容易混淆的样本。
- 改稿动作执行不精准的样本。

新增样本必须同步更新 fixture 和 `tests/test_direct_report_quality_regression.py`，不得把真实内部材料或生成正文提交到仓库。

历史基线、第一阶段实施记录和人工评审见 `docs/history/quality-reviews/`，仅用于追溯，不作为当前规则。
