# 直报生产验收

本文档规定直报能力上线或发生关键改动后的重复验收方法，不记录某一次测试结果。历史测试结论进入月度开发日志或 `docs/history/quality-reviews/`。

## 验收链路

```text
企业微信写作 Bot
  -> app/writing/bot.py
  -> app/platform/app.py
  -> skills/direct_report/
  -> 受限网页、文档和知识库工具
  -> Pydantic AI
  -> 写作任务目录
  -> 企业微信回复
```

## 前置条件

- 使用 `uv sync --locked` 建立项目环境。
- 写作 Bot 和模型配置已写入本机 `.env`。
- 正式用户权限通过 `M_AGENT_PLATFORM_POLICY` 指向本机权限文件。
- 测试材料已获授权；真实材料、任务结果和日志不进入 Git。

配置检查：

```bash
uv run --locked python -m app.writing.bot --check-config
```

检查结果必须脱敏显示 Bot 和模型配置，并确认 Skills、权限文件及 `M-Agent-Files/` 数据目录有效。

## 自动化回归

```bash
uv run --locked pytest \
  tests/test_writing_platform_bot.py \
  tests/test_writing_task_execution.py \
  tests/test_platform_app.py \
  tests/test_platform_wecom_gateway.py \

  tests/test_direct_report_quality_regression.py -v
```

涉及公共文档读取、网页读取、会话或附件交付时，按 `docs/development/testing-and-delivery.md` 补跑对应底座测试。

## 企业微信验收

### 正常新稿

分别使用一条可读公开链接、一段足够长的纯文字和一个获授权的 DOCX/PDF/PPTX 文件发起直报。

通过标准：

- Bot 快速确认收到，话术准确说明是直报任务，不暴露队列或内部任务编号。
- 稍后返回标题、正文和来源；任务状态落入 `M-Agent-Files/tasks/writing/`。
- 直报主体统一使用“微众银行”，不使用全称加简称或“我行”。
- 文件材料只从本次任务目录读取，PPTX 只作为写作素材，不宣称完成视觉审核。

### 材料不足

只发送“帮我写直报”或明显过短的材料。

通过标准：系统追问用户补充链接、文字或文件，不调用模型生成空泛稿件。

### 多轮改稿

完成初稿后，连续提出“只改标题”“压短第二段”“回到上一版再改”等要求。

通过标准：系统沿用正确的直报任务和版本，不误建新 Skill 任务；版本链可恢复，具体局部修改结果按 `TODO-018` 的当前边界如实评估。

### 链接部分失败

同时提供可读链接和不可读链接。

通过标准：系统说明哪些材料无法读取，并询问是使用其余材料继续，还是等待用户粘贴正文；不能因单个链接失败直接丢失整项任务。

### 未授权用户

使用没有 `direct_report` 权限的测试身份发起任务。

通过标准：系统明确拒绝，不读取网页、文件或调用模型。

### 重复与恢复

在测试环境重复投递同一消息，并在处理或交付检查点模拟 Bot 重启。

通过标准：同一消息不生成两份稿件；已成功交付的结果不重复发送；无法确认发送状态时停止自动重发并告警。

### 安全失败

使用不可访问链接、损坏文档或模拟模型失败。

通过标准：用户只收到简短、可操作的提示；详细错误进入受限日志和运维事件，不返回密钥、堆栈、本机路径或用户材料。

## 通过条件

以下条件必须同时满足：

1. 配置检查和相关自动化测试通过。
2. 链接、文字和文件新稿链路至少各验证一次。
3. 多轮改稿、权限拒绝、材料不足和链接失败行为符合当前协议。
4. 重复消息、进程恢复和附件交付不产生重复结果。
5. 任务、日志和用户材料全部位于 `M-Agent-Files/`，没有进入仓库。
6. 审核 Bot 的专项保护测试仍通过。

具体某次验收的功能变化、关键结果和剩余风险写入当月开发日志；不要在本文追加日期、任务编号或测试数量快照。
