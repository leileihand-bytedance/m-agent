# 简报质量回归方法

本文规定单素材和多素材简报如何重复评估。业务写作规则仍以 `skills/writer1/SKILL.md` 和 `skills/writer2/SKILL.md` 为准；本文不保存真实简报正文、任务编号或逐篇评语。

## 样本分层

本机经用户提供的 15 份简报案例用于建立参考池，不进入 Git，也不默认发送给外部模型。样本按用途分为三层：

- A 类：主题集中、事实密度较高、结构清楚、接近可直接报送，用于提炼正向写法。
- B 类：类型典型且有可借鉴部分，但仍需调整结构、信息取舍或表达，用于边界对照。
- C 类：信息密度较低、宣传或会务内容偏多，用于提炼反向约束。C 类题材在业务上仍可能必须报送，不能因此拒绝写作。

仓库内的脱敏等价样本位于：

```text
tests/fixtures/brief_quality_cases.json
```

固定样本覆盖综合成果、机制成果、产品工具、平台合作、标准引领、能力建设、外部认可、活动亮相和专项治理九类，不包含真实材料和真实成稿。

## 自动化检查

```bash
uv run --locked pytest tests/test_brief_quality_regression.py tests/test_brief_revision.py -q
```

自动化测试负责检查：

- 九类简报均有明确的“必须覆盖、主动压缩、推荐结构”类型卡。
- 生成前规划能形成事实台账，并允许语义规划器选择主线、段落和事实编号。
- 正文超过 1200 字时必须触发硬约束。
- 正文新增材料中不存在的数字或“首个、唯一、领先”等强结论时必须触发硬约束。
- 只改标题、只改指定段落、目标篇幅和保留数字等改稿指令不会被扩大为无约束重写。

确定性规则只能识别明确错误，不能独立判断整篇文章是否值得报送、类型选择是否最佳、监管价值是否自然或初稿是否真正可用。

## 人工评测

每次影响简报 Skill、规划器、事实台账、critic、材料补充或改稿行为时，对同一批经授权样本做前后对照。每篇按以下维度记录 `0` 到 `3` 分和一句主要问题：

| 维度 | 判断内容 |
| --- | --- |
| 可用性 | 能否作为正式简报初稿继续修改 |
| 报送价值 | 是否让地方政府和监管部门快速理解近期动态及成果 |
| 事实准确 | 是否忠于材料，数字、评价和因果关系均有依据 |
| 类型适配 | 是否采用对应类型的重点和取舍，而不是套用单一模板 |
| 结构 | 主线是否集中，2 至 3 个核心板块是否清楚 |
| 简报文体 | 是否正式、克制，避免新闻稿、宣传稿和会务报道腔 |
| 修改成本 | 用户需要大改、中改还是小修 |

评分含义：

```text
0 = 不合格
1 = 勉强可用，需要大改
2 = 基本可用，需要中等修改
3 = 可用，只需小修
```

## 执行与反馈

1. 先运行自动化测试，再在获得明确数据传输授权的环境中调用真实模型生成当前版本结果。
2. 前后版本使用同一批样本、同一模型配置和同一评测维度；模型波动与代码差异分开说明。
3. 真实输出、逐篇评分和用户材料只保存在 `M-Agent-Files/evaluations/` 或受限任务目录，不写入 Git。
4. 用户后续改稿要求用于识别高频修改点；只有脱敏且可泛化的模式才转成类型卡、规则、提示词或固定测试。
5. 用户明确确认采用的最终版本可作为候选正向样本；未获得确认时不能把“已生成”推断为“已采用”。
6. 简报的最终采用信号和自动化质量统计尚未建立，继续由公共内容生产反馈闭环待办统一建设，不能用改稿次数简单替代采用情况。
