# M-Agent 功能区规划

## 定位

功能区用于沉淀 M-Agent 的具体业务能力，也就是后续主要要建设的 skills。

功能区不负责企业微信接入、用户权限、文件隔离、工具授权等公共能力。这些属于底座区。

一句话概括：

> 功能区负责“做什么、怎么做、输出成什么样”；底座区负责“谁能用、怎么安全运行、调用哪些工具”。

## 推荐代码目录

当前已建立：

```text
skills/direct_report/
skills/writer1/
skills/writer2/
skills/rewrite/
skills/research_synthesis/
```

后续正式开发时，再继续新增：

```text
skills/
├── direct_report/
├── writer1/          # 简报写作：单素材，已接入新底座
├── writer2/          # 简报写作：多素材，已接入新底座
├── research_synthesis/ # 综合调研：按现成提纲整合多部门素材
├── review/
└── meeting_minutes/
```

`rewrite` 已实现，当前只接收用户直接粘贴的文字，不接收文件或链接；支持“要求在前、原文在后”或“原文在前、要求在后”两种贴法；独立润色 Bot 收到纯原文时会先追问修改方向，不直接调用模型，用户补充要求后再执行；可结合底座会话版本继续修改上一稿。

每个 skill 独立一个目录。不要把多个能力混在同一个 prompt 或同一个脚本里。

## 单个 Skill 的推荐结构

```text
skills/direct_report/
├── SKILL.md
├── config.yaml
├── schema.py
├── workflow.py
├── prompts/
│   ├── draft.md
│   └── revise.md
└── tests/
    └── test_direct_report.py
```

### `SKILL.md`

给模型和开发者看的能力说明，写清楚：

- 这个 skill 什么时候使用
- 输入材料可能是什么
- 具体执行步骤
- 写作风格或审核规则
- 禁止事项
- 自检清单
- 常见失败情况

### `config.yaml`

给底座读取的配置，写清楚：

- skill 名称
- 触发条件
- 允许使用哪些工具
- 是否允许多轮改稿
- 是否需要用户确认
- 输出格式

示例：

```yaml
id: direct_report
name: 直报写作
enabled: true
allowed_tools:
  - web_reader
  - search
  - document_reader
  - word_reader
  - pdf_reader
  - llm_writer
inputs:
  - url
  - file
  - user_instruction
outputs:
  - title
  - body
  - sources
supports_revision: true
```

### `schema.py`

定义输入和输出结构，保证模型输出可检查。输出模型必须优先使用 Pydantic `BaseModel`，方便 Pydantic AI 做结构化输出。

例如直报写作至少应输出：

- 标题
- 正文
- 来源说明
- 是否需要追问

### `workflow.py`

实现这个 skill 的执行流程。

它可以调用底座提供的受限工具，但不能直接访问本机任意文件，也不能绕过底座权限。模型调用应通过底座的 Pydantic AI 执行层完成。

### `prompts/`

存放具体提示词。

建议把初稿、改稿、审校分开，避免一个 prompt 越写越长。

### `tests/`

每个 skill 都要有测试。

测试重点不是证明模型一定写得好，而是验证：

- 能正确识别输入。
- 能调用允许工具。
- 不会调用未授权工具。
- 输出结构符合要求。
- 材料不足时会追问。

## 近期 Skill 规划

### 1. 直报写作

优先级：最高。

原因：

- `skills/direct_report/` 已经是正式直报 skill 目录。
- 它能验证链接读取、写作、企业微信返回的完整链路。
- 它是后续简报、改稿等能力的模板。

第一阶段本地目标已跑通：

```text
用户发链接
  -> 读取网页
  -> Pydantic AI Agent 生成结构化直报初稿
  -> 本地 demo 输出
```

后续增强：

- 优化直报成稿质量。
- 已支持 Word/PDF/PPTX 材料，统一通过底座 `document_reader` 安全解析；PPTX 当前作为写作素材读取，不包含视觉版式审核。
- 支持用户补充要求。
- 已支持上一稿改稿 v1：用户在直报初稿后继续提出“再压缩一点”“突出政策背景”等要求时，系统会基于上一稿修改。
- 企业微信新直报已接入写作专用持久队列：先创建正式 job 和素材快照、返回直报写作已受理提示，再由后台生成并主动发送；同一消息不会重复建稿，worker 重启后从检查点恢复。后台任务编号正常不向用户展示，异常需要人工介入时显示“处理编号”。上一稿改稿仍走原实时链路，不与新稿队列混用。
- 部分链接读取失败时会保留任务材料；用户确认“继续使用已读取素材写”后会忽略失败项继续，不再反复询问同一个选择。
- 支持来源说明和事实自检。
- 接入企业微信入口后的多 skill 路由。
- 优化搜索结果筛选、材料展开和事实自检。

当前必须遵守的直报质量规则：

- 主体称谓全篇直接使用“微众银行”，不得写全称、不得写“以下简称”、不得写“我行”。
- 不得为了显得正式而硬挂政策。案件、活动、获奖、直播、判决等时间节点稿件默认直入主题，不额外注入政策库材料。
- 只有产品、机制、长期服务、政策落地类素材，才优先考虑补充政策背景。
- 具体政策文件名称只能来自用户素材或补充材料；没有明确来源时，不写具体政策名。
- 质量回归样本维护在 `docs/development/direct-report-quality-regression-v1.md` 和 `tests/fixtures/direct_report_quality_cases.json`。

### 2. 简报写作

优先级：高。

当前已安装两个简报 skill：

```text
skills/writer1/   # 单素材简报写作
skills/writer2/   # 多素材整合简报写作
```

当前状态：

- 已导入 `SKILL.md`。
- `writer1` 已导入知识库文件。
- 已创建 `config.yaml`。
- 已实现 `workflow.py`、`schema.py` 和测试。
- 已启用路由，可通过“写简报”“多素材简报”等触发。
- 已接入 `bank_materials`，写作前会自动补充少量高相关微众银行自身背景、产品和数据。
- 已接入共享 `policy_research`，会先判断“能不能挂政策、挂哪条、为什么、摘哪一句”；低层 `policy_materials` 仍保留为兼容材料包工具。
- `wiki` 已从写作链路中下线，不再作为简报写作依赖。
- 已支持上一稿改稿 v1：`writer1` / `writer2` 的初稿生成后，用户可继续要求压缩、补充、调整标题或改变表达重点。
- 已新增共享“简报写作规划”层：单素材简报会先提炼核心主线，多素材简报会先判断素材关系、统一主题和整合重心。
- 已新增简报确定性质量检查：标题连接方式、首次主体称谓、正文列表式结构等高价值规则会在生成后兜底检查。
- 已新增简报语义审查与一次自动重写：对明显新闻稿腔、拼盘式整合、改稿后风格漂移等问题，会带反馈重写一次。
- `writer2` 遇到明显弱关联素材时，会直接提示拆分或补充材料，不再强行整合。
- 简报当前真实口径已明确：报送对象是深圳市金融办、南山区、前海管理局、深圳人行、深圳金监局等地方政府和监管部门，主要用于展示微众银行近期动态及成果。
- 简报篇幅与直报不同：正常控制在 `1000` 字左右，最长不超过 `1200` 字。
- 已按经授权样本沉淀第一版简报类型分类：综合成果型、机制成果型、产品工具型、平台合作型、标准引领型、能力建设型、外部认可型、活动亮相型、专项治理型。
- 写作 Bot 已新增短任务组装 v1：用户可以先说“帮我写简报”，再连续发送多个链接、文字或文件，最后回复“开始写”；也可以先发材料，再补一句“写简报”。入口层会根据材料数量选择 `writer1` 或 `writer2`，skill 本身不负责维护企业微信聊天状态。
- Word、原生文字 PDF、扫描 PDF 和 PPTX 已通过统一文档服务进入 `writer1` / `writer2`；待组装文件可在 Bot 重启后恢复，扫描 PDF 只对识别出的扫描页按需 OCR。
- 企业微信新简报已接入与直报共用的写作持久队列：`writer1`、`writer2` 分别登记任务类型，共用单用户串行、处理/会话/发送检查点和主动文字交付。重复文件消息会在暂存阶段去重，避免一份素材被误计为两份后错路由到 `writer2`。
- 用户有直报或简报初稿在途时，入口暂不接收下一批写作材料或改稿，明确提示等待初稿；这样旧任务完成时不会清掉用户刚发的新材料。收到初稿后，原有多轮改稿能力继续生效。
- 当前还需要继续优化成稿质量和多素材体例。
- 简报主体称谓与直报不同：正文首次出现主体时写“深圳前海微众银行（以下简称“我行”）”，后文统一写“我行”。

与直报写作不同，简报可能更强调：

- 多材料综合
- 结构清晰
- 信息密度
- 可读性
- 面向地方政府和监管部门展示近期动态及成果

可以复用直报写作的基础工具，但 prompt、结构和输出标准应独立。

### 3. 智能审核

优先级：当前主线。

现有 `app/review/` 已经能独立运行，支持三种自动识别的 Word 内容审核、一种显式触发的独立格式审核、单份 PPTX 低级错误审核，以及多文件联合审核 v1：

- **内参周报**：原有 13 条规则，两阶段 LLM 审核
- **半月报**：独立规则库，含时间范围、一级标题归属、行内领导职务与排序规范
- **通用审核**：代码规则 + 分段语义审核 + 通篇逻辑校对，返回带原文定位和批注的 Word 文档
- **HTML 文字审核**：单个 `.html/.htm` 只提取静态可见标题、段落、列表和表格行，复用通用规则检查低级错误、语病和页面内部数据一致性，只返回审核消息；网页 PPT 按 `class="slide"` 的页面顺序显示错误页码，普通 HTML 回退到段落位置
- **公文格式审核**：用户可在发送 `.docx` 前或后 30 分钟内明确说“格式审核”，系统按常用公文模板检查实际字体、字号、加粗、标题层级、对齐、缩进、行距和页面设置，不接入通用内容审核
- **PPT 低级错误审核第一阶段**：用户直接发送单份 `.pptx`；独立业务包检查可编辑文本框、表格和可读图表文字中的低级错误，以及同一份 PPT 内部主体、时间和指标口径一致的明确矛盾，只返回“页码 + 原文 + 事实问题”的分段文字，不提供修改建议
- **多文件联合审核 v1**：用户直接连续发送 2 至 5 份 `.docx` 即自动联合审核，无需前置或结束指令；系统不按上传顺序默认正文，综合文件名和正文附件引用识别，证据不唯一时才要求用户指定主文件

执行方式当前按“单项/多文件”分开：纯文字、单个通用 Word、内参、半月报、公文格式、单个静态 HTML 和单份 PPTX 已经接入审核专用持久队列。文件到达时即时确认收件；HTML 与 PPTX 下载和大小校验后分别直接入队，不进入 Word 单/多文件归集，不参与公文格式或联合审核。HTML 只提取静态可见文字并复用通用审核，不执行 JavaScript 或联网资源；存在至少两个可见段落时，短于 200 字也做通篇数据一致性检查。网页 PPT 的消息和归档报告按 `slide` 容器顺序显示页码，普通 HTML 显示段落位置，不计算浏览器打印分页。复杂 CSS 可见性、动态渲染、图片/图表和 HTML 版式不在当前范围。

PPT 包可以复用底座解析、任务、模型配置和企业微信发送，但不导入通用、内参、半月报、公文格式或多文件审核引擎。PPT 模型候选必须命中指定页、对象和逐字原文。名称不一致还必须提供同一专有名称的两种不同原文写法；代码只在忽略大小写、宽窄字符和名称内常见分隔符后两边仍是同一名称时保留，因此不会把单独出现的 `CyclOne`、不同项目名称、相同重复脚注、脚注标记兼容变体或常见注释语句当作名称不一致，反向重复上报也只保留一条；`C++` 这类符号结尾名称仍可校验。跨页矛盾还必须命中双边原文并满足同主体、同时间、同指标口径；代码会硬拦截原文明示的不同年份、时间范围、单位和目标/实际状态。用户可见的问题说明不透传模型自由文本，只使用代码维护的类别事实说明，从结构上排除修改建议。语言审核常规批次约 3000 字符；模型明确因输出上限停止时，按 PPT 对象边界自动拆半、分别审核并合并结果，普通网络或 JSON 异常不进入该分支。语言分批和全篇一致性阶段各自保存输入摘要断点，恢复时不重复调用已完成顶层批次；图表读取不完整会在文字结果中显示页码提示。PPT 第一阶段明确不检查图片文字、演讲者备注、外部事实、视觉版式或多份 PPT 之间的一致性；跨 PPT 属于第二阶段。

首份经授权 20 页真实 PPT 已完成端到端实测。针对实测中的单边名称、不同项目名称和相同脚注误报，修复后的已保存候选回放只保留第 1 页明确占位符；本轮隔离回归为全仓 `1058` 项、PPT 与入口联合 `173` 项、PPT reviewer `36` 项、审核 Bot 保护测试 `9` 项通过。更多真实样本的漏报、页码、图表读取和长结果展示仍待继续验收，不能据此宣称第一阶段质量已经稳定。

纯文字和文件后补的格式要求只返回一次简短处理确认。`“帮我查一下格式”`等口语要求与“格式审核”“按公文格式检查”统一识别为独立公文格式审核；“也做一下文字审核”“再做一下内容审核”等对刚发文件的完整短操作指令不会另建纯文字任务。完整任务编号只在后台用于恢复、幂等和排障，异常需要人工介入时才显示“处理编号”。同一企业微信消息不会重复创建任务，最终结果只发送一次，发送状态无法确认时会停止自动重发并告警，后台 worker 异常会自动重启。Word 单文件仍可在 30 分钟内追加独立格式审核。多文件联合审核继续走原来的独立流程。HTML 和 PPTX 都只返回企业微信审核消息，不生成标记文件。

通用审核当前已经支持：

- 占位内容、编号跳号、空标题、附件引用、附件编号名称错配、无效日期、显式起止时间冲突和术语错写等确定性检查。
- 引号配对规则会识别英文单词内部和复数所有格末尾的弯撇号，`People’s`、`people‘s`、`students’` 等正常英文写法不会按中文单引号不成对报错。
- 错别字模型使用零温度和“原文错误片段应为正确写法”的统一输出方向；模型生成的错别字候选无论文档篇幅都结合完整段落和前后文做两次严格语义复核，只有一致确认才保留，避免把相邻但分属不同语法成分的字拼成错词。代码词表等确定性错字仍直接保留。证据自相矛盾的候选同样进入两次定向复核。分块会把“数字序号 + 短标签”与后续说明保留在同一批次，避免 `Fortune` 位于块尾而 `Forbes` 上下文完整所造成的不一致误报；不再按短英文标签做泛化过滤。标点后真实多余空格仍缩小到精确字符并使用正确标点名称说明。
- 错别字、名称、语病、标点、内容不完整、重复内容和跨段逻辑矛盾等语义检查。
- `200` 至 `100000` 字文档的通篇逻辑校对；`20000` 字及以上长文使用双扫描和候选复核降低漏报、误报。
- 标注时优先用完整错误原文在全文定位，再用内部段号辅助；自动位置无法确认时，批注仍提供可搜索原文供用户手工定位。
- 真实文件质量评测工具按二进制和正文近重复双重去重，自动选取问卷、附件引用、长文、表格和常规材料，复用 Bot 主链路生成标注文档、逐条评分表，并记录耗时、模型请求、请求失败和未恢复审核阶段。

多文件联合审核 v1 当前支持：

- 文件到达即回复确认；后台默认使用 8 秒静默窗口自动分流，只有 1 份时继续走原单文件审核，2 至 5 份时进入联合审核。
- 每份文件复用原有内参、半月报或通用内容审核，再追加跨文件检查。
- 确定性检查正文附件引用与实际上传文件的缺失、重复、未引用和编号名称错配。
- 通篇模型检查名称、日期、数量、金额、状态、统计范围和要求冲突；每条意见必须同时命中两份文件中的真实原文证据，否则丢弃。
- 待组装文件按入口和用户隔离并持久化，默认 30 分钟过期；旧的“联合审核”“开始审核”指令仍兼容，但默认不需要。一次任务统一归档并按文件返回标注文档。
- 当前仅支持 `.docx`，且组装接口仍在旧审核模块中；PDF、Excel、PPT 和底座级公共组装后续推进。

当前审核主线按以下顺序推进：

### 4. 改稿与润色

优先级：中。

这里要区分两类能力：

```text
上一稿改稿：
  用户先让 M-Agent 写直报/简报
  -> M-Agent 输出初稿
  -> 用户继续说“再压缩一点”“标题正式一点”
  -> 系统基于上一稿修改

独立改稿 / 润色：
  用户直接丢一篇外部初稿
  -> 提出修改要求
  -> 系统按要求重写、润色或审校
```

当前已完成的是两部分：

1. **上一稿改稿 v1**
由底座的 `ConversationStore` 记录同一用户、同一入口的当前活跃稿件、版本链和修改要求，并由平台 `intent` 模块判断“改当前稿 / 开新任务 / 追问”。该能力由 `direct_report`、`writer1`、`writer2` 共用。

2. **独立 `rewrite` skill v1**
已新增 `skills/rewrite/`，第一版只处理“用户直接贴文字 -> 按要求润色 -> 返回润色后正文 + 修改说明”，并支持继续对话式修改。直接贴文字时，既支持“帮我润色这段：……”这类要求在前的写法，也支持“先贴原文，再单独补一句‘帮我整体润色一下’”。独立润色 Bot 收到只有原文、没有修改方向的消息时，会先保存原文并追问一次；用户可补充具体要求，或回复“按默认方式润色”。

现已增加独立企业微信材料润色 Bot。该入口在注册表层只加载 `rewrite`，不会进入直报、简报、综合调研或审核；只接收直接粘贴的文字，文件和网页链接会提示用户改为粘贴原文。独立入口使用单独任务、待确认原文和会话目录，因此不会继承原写作 Bot 的活跃稿件。待确认原文默认 30 分钟有效，Bot 重启后可恢复，成功处理或用户取消后清除。

`rewrite` 已条件式接入本地 `bank_materials`：只有原文明显出现微众银行、微业贷、微粒贷、微贸贷等信号，或“我行/本行”与微众业务语境同时出现时，才检索少量高相关语料。语料以 `verification_reference` 角色进入模型上下文，只用于复核机构名、产品名和已有标准表述；复核通过不等于允许补充，不能把原文没有的数据、日期、荣誉或新事实写入润色稿。普通材料完全不调用微众银行信息库，实质口径冲突必须追问用户，不能静默改数。

二者必须严格切开：

- 用户继续说“再压缩一点”“标题再稳一点”，沿着原直报/简报上一稿继续改；
- 用户明确发“帮我润色这段：……”这类带新正文的新任务，即使当前存在活跃直报/简报会话，也应新开 `rewrite`，不能误改旧稿。

`rewrite` v1 的边界：

- 只支持用户直接贴文字
- 不读取链接、不读 Word/PDF
- 不联网补资料
- 微众相关材料可读取本地微众银行信息库做名称和标准表述核对，但不补充新事实
- 默认只优化表达，不新增事实
- 对明显残句或 `XX` 占位，仅做最小必要补全；无法确定时追问

该能力后续如要扩展为“外部初稿重组、压缩、扩写、改体例”，再在 `rewrite` 上继续演进，而不是并入直报/简报 skill。

改稿需要知道：

- 原文是什么
- 上一次由哪个 skill 生成
- 用户这次要求怎么改
- 哪些事实不能改

### 5. 综合调研材料整合

当前已新增独立 `skills/research_synthesis/`，用于“用户上传一份现成调研提纲和若干部门素材，系统先解释提纲约束，再形成可回溯的综合调研初稿”。

它与 `writer2` 的边界是：

```text
writer2：
  多份素材 -> 自行判断关联性 -> 提炼统一简报主题和结构

research_synthesis：
  唯一现成提纲 -> 判断覆盖方式 -> 证据台账 -> 跨部门归并 -> Word 初稿
```

第一版已实现：

- 支持 DOCX、原生文字 PDF、PPTX 和补充文字，文件仍由底座统一文档服务安全解析。
- 用户明确点名某文件为提纲时优先采用；否则优先采用唯一使用“调研提纲”等明确名称的总提纲，并结合正文中的回复、答复和反馈特征排除部门材料。
- 无法唯一识别提纲时追问，不按上传顺序猜测；追问期间保留原文件和任务状态，用户回答后自动续跑。
- 提纲至少要配 1 份部门素材；任一文件读取失败时暂停整合，避免静默缺项。
- 写正文前先判断提纲类型和覆盖方式：问卷、检查清单和报告骨架通常逐项覆盖；政策事项目录按本次任务相关性和可用证据选择主题。无法可靠判断时追问，不静默套用任一模式。
- 结构化台账按提纲问题记录事实、规范化部门来源、材料位置、时间、指标口径、单位、缺口、冲突和图片提醒，并区分原文事实、派生计算、图片候选和本次材料外缺失信息。
- 逐项覆盖模式保留全部必答主题和顺序，模型漏项时由确定性检查补回并提示人工核对；选择性覆盖模式只保留有可用证据的选定主题，同时过滤模型自行加入的未选事项。
- 同类事实去重；只有对象、时间、单位和口径一致，且台账记录原值、运算和结果时，派生合计才可进入正文。空缺章节标注“【材料待补充】”，冲突口径标注“【口径待确认】”。
- 第一阶段继续在综合事实段末保留一次“【来源：XX部、YY部】”，来源标签由工作流规范化，不再展示长文件名或在段首反复标注。
- 台账中的来源标签必须匹配本次上传材料；缺少合法来源的事实段标记“【来源：待核对】”。正文数字必须能在可用证据或核定推导中找到，不能追溯时标记“【来源待核对】”。
- DOCX 素材中的图片不做 OCR、不猜测内容、不嵌入汇总稿；统一文档服务保留原位置提醒，正文在对应小节保留人工评估提示，连续的同部门图片提醒合并计数。
- 一级、二级标题确定性统一为“一、”“（一）”；明确的三级短标题可以保留 `1.`，不再机械改成“一是”。
- 输出采用审核模块现有公文基本格式规则的 Word 初稿，并由企业微信作为文件返回；标题后和文末分别保留开头、结尾待用户补充备注。
- 不联网补事实，不套用简报约 1000 字、监管展示口径，也不在第一版同时做宣传化润色。

当前边界：

- 沿用写作入口现行上限：一次最多 10 份文件、总计不超过 20MB。
- 扫描 PDF 已按需 OCR；识别失败时会保留具体待处理页，不能把未识别内容当成已读取。
- 长文档当前使用统一文档服务的均衡取样；逐条穷尽超长材料仍需建设“分块映射提纲—分章节生成—总稿合并”流程。
- 第一版已输出 Word 初稿，但暂不生成报告开头和结尾、不嵌入图片，也暂不支持基于该稿的独立润色阶段。
- 当前长期回归包含三类脱敏等价场景：选择型政策目录、逐项问卷缺项、派生数字与图片/外部候选混合证据；真实案例原文和业务数据不进入仓库。

后续增强统一由 `TODO-026` 维护，不并入 `writer2` 的简报质量待办。

### 6. 深银协动态

- 时间范围严格以用户要求为准：用户必须明确月份和上半月/下半月；未明确时先追问，答复前不搜索、不生成。追问会保留原任务，用户回答后仍由深银协 Skill 继续执行。
- 通过 ToolGateway 调用 DeepSeek 原生 Web Search，检索 `media_sources.yaml` 登记的权威媒体（如人民网、深圳特区报等）发布的微众银行相关报道；搜索结果只建立候选池，不直接作为正文。

- 通过 `web_reader` 读取正文后，代码先执行日期、域名、正文可读性和逐字段落校验，再由 DeepSeek 结构化判断是否属于可报送的微众银行正面新闻或成果，以及应保留全文、摘编或拒绝。
- 人民日报旧版网页的固定错误 `publishdate` 不再直接采用；仅在 `paper.people.com.cn` 页面内期号目录与正文路径互相印证时使用期号日期。转载标题末尾的媒体站点名先移除再比较，避免同稿因“经济观察网/21世纪经济报道”等尾缀不同而重复入选。
- 完整专题稿优先；两轮信源仍不足时，综合稿只能摘取原文中可独立成立的微众银行相关段落，允许调整标题，但不能全面改写或补充事实。Word 会同时标注原报道标题、原文链接和摘编说明。
- 优先选取 3 篇；不足时降级到 2 篇或 1 篇，0 篇时返回明确提示且不生成 Word。
- 已接入 Word 模板生成：存在 `skills/shenyinxie_news/assets/shenyinxie-template.docx` 且含约定占位符时按模板替换，否则按现有公文格式规则生成兜底文档；生成文件通过 `output_file` 返回，供写作 Bot 作为附件回传。

触发词包括“深银协动态”“生成深银协动态”“整理深银协动态”“深圳银行业协会动态”。该能力由写作 Bot 入口直接识别，不依赖用户提前提供素材；如触发语未说明月份和上/下半月，先进入一次追问。

当前边界：

- 模型负责报送价值、核心主体和摘编可用性判断，代码负责不可越过的来源、日期、正文和原文逐字校验；事件去重仍以确定性规则为主。
- 模板文件不进入 Git；占位模板含约定占位符，真实案例模板替换同名文件即可，无需改代码。
- DeepSeek 搜索会产生额外模型 Token 和搜索调用；真实入选仍以原文页面核验为准。

## Skill 开发工作流

后续你可以用自然语言让 Codex / Claude Code 开发：

```text
新增一个“政策简报写作”skill。
输入是网页链接或 Word/PDF/PPTX 文件。
输出包括标题、摘要、正文、来源。
风格正式、简洁，适合向地方政府和监管部门报送。
请按现有 skill 模板实现，并补测试。
```

AI 编程工具应按以下顺序处理：

1. 读取现有 skill 模板。
2. 新建 skill 目录。
3. 写 `SKILL.md`。
4. 写 `config.yaml`。
5. 写输入输出 schema。
6. 写 workflow。
7. 写测试。
8. 跑测试。
9. 更新相关文档。

## 功能区开发原则

1. 一个 skill 只做一类业务能力。
2. 写作规则放在 skill 中，不放在底座代码中。
3. 工具权限写在 `config.yaml` 中，由底座执行。
4. 每个 skill 必须有测试。
5. 新增 skill 默认不开放给所有用户，需要显式配置权限。
6. 用户需求不属于已有 skill 时，不让模型自由发挥，应拒绝或追问。

## Skill 完成标准

一个 skill 至少满足：

1. 有清晰的 `SKILL.md`。
2. 有明确的输入输出结构。
3. 声明了允许使用的工具。
4. 有可运行 workflow。
5. 有测试。
6. 能被底座注册表发现。
7. 能通过企业微信入口调用。
8. 不访问未授权文件和工具。

## 与底座区的边界

功能区可以写：

- 直报怎么写
- 简报怎么写
- 审核规则是什么
- 改稿时遵守什么风格
- 输出格式是什么

功能区不写：

- 企业微信 SDK 连接
- 用户白名单
- 任务目录创建
- 本机文件权限
- 工具全局注册
- 系统日志
- API 密钥读取

这些都属于底座区。
