## 一、定位

企业微信经验沉淀入口是第一阶段优先落地的子项目。

它不是完整写作工具，也不是通用聊天机器人。它只解决一个核心问题：用户日常把文档、讨论记录、修改意见、定稿对比等材料发给企业微信智能机器人，后台自动提炼经验沉淀建议，并通过企业微信返回给用户确认。用户确认后，系统再写入长期知识库。

## 二、为什么从这里开始

完整写作流程需要较多 Agent 配合，且依赖公司知识、政策知识和领导风格知识。如果这些知识一开始为空，直接做写稿工具容易效果不稳定。

经验沉淀入口可以先解决知识积累问题，尤其是领导风格积累问题。它能让用户在日常工作中低成本把材料交给 AI 分析，而不是每次手工整理。

## 三、技术路线

第一版采用：

```text
成熟企业微信长连接 SDK + 自建经验沉淀业务逻辑
```

不建议第一版直接将 OpenClaw、Hermes、LangBot 等大框架作为主系统，也不建议从 WebSocket 协议层完全自建。

### 1. 为什么不直接上大框架

OpenClaw、Hermes、LangBot 等框架适合通用机器人、多渠道接入、长期运行和复杂 Agent 平台。它们可以解决很多通用问题，但也会带来额外框架复杂度。

本项目第一阶段的核心不是通用聊天，而是一个非常具体的业务闭环：

1. 收到日常材料。
2. 识别材料属于哪位领导、哪类经验。
3. 提炼风格观察和写作规则。
4. 生成可确认的沉淀建议。
5. 等用户确认。
6. 写入项目知识库。

这个流程对“确认后才入库”的要求很强。如果直接把它放进大框架，可能会增加定制和排查成本。

### 2. 为什么不完全自建

企业微信智能机器人长连接涉及认证、心跳、断线重连、文件下载解密、消息类型处理、主动回复等底层细节。

这些能力不应该从零写。第一版应复用成熟 SDK 来完成企业微信接入层，把精力放在经验沉淀业务逻辑上。

### 3. 推荐边界

企业微信接入层可以替换，经验沉淀核心逻辑不要绑定在某个框架里。

也就是说：

```text
入口层：可以用 SDK，未来也可以替换为 OpenClaw / Hermes / LangBot
核心层：本项目自建，沉淀为 agents、templates、knowledge、runs
```

## 四、最小闭环

第一版只做领导风格沉淀。

用户在企业微信里发送：

```text
@M-Agent 沉淀领导风格：张总
```

然后上传 Word、PDF、TXT、Markdown、聊天记录或修改意见。

后台流程：

1. 接收企业微信消息。
2. 判断当前任务类型是“领导风格沉淀”。
3. 接收并下载文件。
4. 解析文件内容。
5. 保存原始材料到：

```text
knowledge/leader-style/张总/inbox/
```

6. 经验沉淀 Agent 生成提炼建议，保存到：

```text
knowledge/leader-style/张总/extracted/
```

7. 机器人把摘要返回给用户确认。
8. 用户确认后，系统更新：

```text
knowledge/leader-style/张总/style-card.md
knowledge/leader-style/张总/accepted-expressions.md
knowledge/leader-style/张总/avoid-list.md
knowledge/leader-style/张总/update-log.md
```

## 五、用户交互示例

用户：

```text
@M-Agent 沉淀领导风格：张总
```

机器人：

```text
收到。请继续发送本次用于沉淀“张总”风格的材料，可以是修改意见、会议纪要、定稿对比或历史稿件。
```

用户上传文件后，机器人返回：

```text
已收到 2 个文件，正在提炼张总的领导风格观察。
```

分析完成后，机器人返回：

```text
本次建议沉淀 5 条：

1. 成绩表达偏低调务实，慎用“领先”“突破”等表述。
2. 政策建议宜使用“建议进一步研究”“可考虑完善”等稳妥表达。
3. 更偏好“阶段性成效”“持续推进”“积极探索”类表达。
4. 对涉及监管和风险的内容，倾向强调稳健合规。
5. 不建议沉淀本次项目数据口径，可能只适用于本篇材料。

请回复：
- 确认全部
- 确认 1、3、4
- 不入库
- 修改第 2 条为……
```

用户确认后，机器人返回：

```text
已更新张总领导风格卡，并记录本次更新来源。
```

## 六、后台模块

第一版后台建议拆成以下模块：

1. 企业微信长连接模块：负责连接企业微信、接收消息、发送回复。
2. 消息路由模块：识别用户指令，例如领导风格沉淀、确认入库、不入库。
3. 文件处理模块：下载文件、解密文件、提取文本。
4. 经验沉淀 Agent：根据模板生成沉淀建议。
5. 确认处理模块：解析用户确认指令。
6. 知识库写入模块：将确认后的经验写入 `knowledge`。
7. 日志与任务记录模块：保存每次处理记录，便于追溯。

## 七、建议支持的文件类型

第一版建议支持：

1. Word：`.docx`
2. PDF：`.pdf`
3. 文本：`.txt`
4. Markdown：`.md`

后续再考虑：

1. 图片 OCR。
2. 语音转文字。
3. Excel 表格。
4. 企业微信聊天记录自动导出解析。

## 八、安全与权限

第一版必须考虑以下边界：

1. 只允许白名单用户触发经验沉淀。
2. Bot ID 和 Secret 不能写入公开文件。
3. 原始文件和长期知识库要放在受控目录。
4. 经验沉淀 Agent 不能未经确认直接写入长期知识库。
5. 每次写入都要记录来源、时间和确认人。
6. 涉及敏感数据时，优先保留为任务记录，不直接沉淀为长期知识。

## 九、后续升级方向

第一版跑通后，可以逐步扩展：

1. 从领导风格沉淀扩展到写作规则沉淀。
2. 从日常材料沉淀扩展到正式写稿任务复盘。
3. 增加公司知识和政策知识的候选沉淀。
4. 增加历史经验检索，让写稿流程自动参考已确认经验。
5. 如未来需要多渠道、完整权限体系和长期运行监控，再评估接入 OpenClaw、Hermes、LangBot 等框架。

## 十、第一版成功标准

第一版成功标准如下：

1. 用户可以通过企业微信机器人发送文件。
2. 后台可以收到并解析文件。
3. 系统可以生成结构化的领导风格沉淀建议。
4. 用户可以通过企业微信确认哪些建议入库。
5. 系统只把用户确认后的内容写入长期知识库。
6. 每次更新都有来源记录，后续可追溯。
